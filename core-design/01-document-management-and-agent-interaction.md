# 核心设计 01：文档管理体系与 Agent 交互协议

> **文档定位**：核心设计思路沉淀，不涉及具体实现。目的是确保后续执行设计不偏离方向。
>
> **创建日期**：2026-06-19

---

## 1. 设计目标

解决 AI Agent 与人协同开发时的三个核心问题：

| 问题 | 表现 |
|------|------|
| Agent 失忆 | 每次启动不知道项目历史、当前状态、下一步该做什么 |
| 文档散落 | 迭代产物分散在各处，做完就丢，无法回溯 |
| 外部事件丢失 | Git Hook、CI 等外部触发的事件，Agent 无法感知 |

---

## 2. 三层信息模型

```
┌─────────────────────────────────────────────────────┐
│  第一层：AI Context                                  │
│  Docs 的摘要 + 索引，Agent 日常工作记忆               │
│  篇幅 < 2000 字，Agent 日常只读这个                    │
├─────────────────────────────────────────────────────┤
│  第二层：project/                                    │
│  项目当前静态状态，唯一的现状真相源                     │
│  只在迭代归档时更新，保证稳定性                        │
├─────────────────────────────────────────────────────┤
│  第三层：iterations/ + archive/                      │
│  变更历史，按迭代维度组织                              │
│  一个迭代 = 一个目录 = 全部资料集中存放                 │
└─────────────────────────────────────────────────────┘
```

### 各层关系

- **Agent 日常读取路径**：AI Context → 需要细化时，通过索引跳转 project/ 或 archive/
- **信息流动方向**：迭代进行中 → 归档时合并到 project/ → 同步更新 AI Context
- **更新频率**：AI Context（每次归档/变更时）> project/（仅归档时）> iterations/（迭代进行中持续）

---

## 3. AI Context

### 3.1 定位

**Docs 的摘要 + 索引**，不是全量复制。

- **摘要**：Agent 快速了解项目核心信息（概览、当前状态、关键约束）
- **索引**：需要深入时，通过路径跳转到具体文档

### 3.2 内容结构

| 区块 | 内容 | 作用 |
|------|------|------|
| 项目概览 | 项目名、技术栈、核心依赖、模块清单 | Agent 知道"这是什么项目" |
| 当前状态摘要 | 已上线模块、进行中迭代、关键约束 | Agent 知道"现在到哪了" |
| 模块索引 | 每个模块的现状文档路径 + 状态 | Agent 按需深入 |
| 迭代历史摘要 | 每个归档迭代的摘要 + 路径 | Agent 了解演进脉络 |
| 同步信息 | 最后同步时间、同步详情引用 | 同步 Skill 的断点续传依据 |

### 3.3 维护方式

- 由专门的同步 Skill 负责维护
- **增量更新**：每次同步记录文件快照（hash + mtime），下次只处理变更部分
- 同步元数据独立存储，不混入 AI Context 正文

---

## 4. project/ — 项目当前状态

### 4.1 定位

项目现状的**唯一真相源**。只在迭代归档时更新，迭代进行中保持不变。

### 4.2 文档拆分（1 + N 结构）

| 文档 | 职责 | 视角 |
|------|------|------|
| `architecture.md` | 系统架构全景：模块关系、技术选型、部署拓扑 | 全局 |
| `data-model.md` | 当前有效 ER 关系、表结构概览 | 全局 |
| `api-contracts.md` | 当前有效接口总览 | 全局 |
| `modules/*.md` | 每个模块的业务现状：规则、流程、边界 | 模块级 |

**拆分原则**：
- 前三个是**跨模块的全局视角**
- `modules/` 是**单模块视角**，一个模块一个文件
- Agent 查模块细节时，通过 AI Context 索引直达

---

## 5. 迭代目录

### 5.1 目录命名

```
{日期}_{需求名称}_{版本号}
```

示例：`2026-06-19_用户认证_v1.0`

- 日期在前，保证时间排序
- 需求名称提供语义
- 版本号标记迭代轮次

### 5.2 目录结构

一个迭代是一个**自包含的项目包**，所有资料集中存放：

```
2026-06-19_用户认证_v1.0/
├── _meta.yaml          # 迭代元信息（状态、时间、影响模块、摘要）
├── prd.md              # 产品需求文档
├── tech-design.md      # 技术设计文档
├── tasks.md            # 任务分解与状态
├── ddl-changes.md      # DDL 变更记录
├── api-changes.md      # 接口变更（新增/修改/废弃）
└── review-notes.md     # 评审记录、决策记录
```

**核心原则**：一个迭代内的所有资料集中存放，不散落到项目其他地方。

### 5.3 迭代状态流转

```
in_progress → completed → archived
```

| 状态 | 含义 | 目录位置 |
|------|------|---------|
| `in_progress` | 进行中 | `iterations/` |
| `completed` | 开发完成，待归档 | `iterations/` |
| `archived` | 已归档 | `archive/` |

---

## 6. 归档流程

迭代完成后的归档操作，信息汇入两个方向：

| 归档动作 | 源 | 目标 | 说明 |
|----------|---|------|------|
| DDL 合并 | `ddl-changes.md` | `project/data-model.md` | 追加新表/字段，标记废弃 |
| API 合并 | `api-changes.md` | `project/api-contracts.md` | 新增/更新/废弃接口 |
| 业务合并 | `prd.md` + `tech-design.md` | `project/modules/*.md` | 更新模块业务规则 |
| 架构合并（如有） | `tech-design.md` | `project/architecture.md` | 架构级变更时 |
| 目录迁移 | `iterations/xxx` | `archive/xxx` | 移动目录 |
| AI Context 同步 | 归档信息 | `AI-CONTEXT.md` | 增量更新摘要和索引 |

**关键约束**：
- 归档是一次性操作，归档后迭代目录变为只读
- `project/` 文档只在归档时更新，迭代进行中不动

## 7. Git 分支绑定

### 7.1 策略

**一个 Git 分支绑定一个迭代**，分支名 = 迭代目录名。

代码生命周期 = 文档生命周期，归档 = 代码合并 + 文档归档。

### 7.2 Git Hook 触发点

| Hook | 触发条件 | 产生的事件 | Agent 响应 |
|------|---------|-----------|-----------|
| `post-checkout` | 从 main 创建新分支 | `branch-created` | 创建迭代目录、初始化 _meta.yaml |
| `post-merge` | 分支合并到 main | `branch-merged` | 执行归档流程、同步 AI Context |

## 8. 脚本-Agent 交互协议

### 8.1 两种交互模式

| 模式 | 触发方 | 通信方式 | 特点 |
|------|--------|---------|------|
| **同步** | Agent 主动调用脚本 | 脚本 stdout 返回结果 | Agent 直接读取处理 |
| **异步** | 外部事件（Git Hook/CI/人工） | 写入事件信箱文件 | Agent 下次启动时扫描补读 |

### 8.2 事件信箱（Event Inbox）

```
.harness/inbox/               # 待处理事件
.harness/inbox-processed/     # 已处理事件（留痕审计）
```

- 事件以 JSON 文件形式存入 `inbox/`
- Agent 启动时扫描 inbox，按优先级处理
- 处理完的事件移入 `inbox-processed/`

### 8.3 事件文件结构

```json
{
  "id": "evt_唯一标识",
  "type": "事件类型",
  "priority": "high | normal | low",
  "source": "来源标识",
  "timestamp": "ISO8601",
  "payload": { "...事件数据..." },
  "required_actions": ["Agent 必须执行的动作"],
  "suggested_actions": ["Agent 可酌情执行的动作"],
  "context": "人类可读的事件描述"
}
```

**关键字段**：

| 字段 | 作用 |
|------|------|
| `required_actions` | 硬流程，Agent 必须执行（如归档） |
| `suggested_actions` | 软建议，Agent 可根据上下文判断 |
| `payload` | 事件数据，不同类型结构不同 |
| `context` | 帮助 Agent 理解事件背景 |

### 8.4 Agent 发现事件的方式

1. **主动扫描**：Agent 启动或切换上下文时，执行 inbox-check Skill 扫描 `.harness/inbox/`
2. **实时通知**（增强）：脚本执行时如果 Agent 在线，额外发送通知；失败不阻塞，inbox 文件保证不丢

### 8.5 已识别的事件类型

| 事件类型 | 触发源 | 优先级 | 必须动作 |
|----------|--------|--------|---------|
| `branch-created` | Git Hook | high | 创建迭代目录、初始化元信息 |
| `branch-merged` | Git Hook | high | 执行归档流程 |
| `commit-pushed` | Git Hook | low | 更新任务状态（如有） |
| `ci-failed` | CI 系统 | high | 分析失败原因、建议修复 |
| `pr-review-requested` | 代码平台 | normal | 执行代码审查 |

## 9. 关键设计决策记录

| # | 决策 | 选择 | 理由 |
|:-:|------|------|------|
| D1 | AI Context 定位 | 摘要 + 索引（非全量） | 控制篇幅，Agent 读得快，深入时按需跳转 |
| D2 | 迭代目录命名 | 日期 + 需求名 + 版本号 | 时间排序 + 语义清晰 |
| D3 | 迭代资料管理 | 集中存放，不散落 | 找资料不用满世界翻 |
| D4 | project/ 拆分 | 1 + N（3 全局 + N 模块） | 全局视角和模块视角分离 |
| D5 | Git 分支策略 | 一个分支 = 一个迭代 | 代码生命周期 = 文档生命周期 |
| D6 | 异步事件传递 | 文件信箱（inbox） | Agent 不在线也不丢信息，简单可靠 |
| D7 | 脚本建议强度 | required + suggested 双字段 | 硬流程必须执行，软建议保留灵活性 |
| D8 | AI Context 同步 | 增量更新 + 快照元数据 | 避免每次全量重建，高效且可追溯 |
| D9 | 同步 Skill 拆分 | 同步 Skill + 归档 Skill 分离 | 同步高频可复用，归档低频且流程不同 |

## 10. 信息流转全景

```
                    ┌──────────────┐
                    │  外部事件源   │
                    │ Git Hook/CI  │
                    └──────┬───────┘
                           │ 写入
                           ▼
                    ┌──────────────┐
                    │  .harness/   │
                    │    inbox/    │──── Agent 启动扫描 / 实时通知
                    └──────────────┘
                           │
                           ▼
                    ┌──────────────┐
                    │   AI Agent   │◄──── 日常读取 AI Context
                    └──────┬───────┘
                           │ 创建/更新
                           ▼
                ┌─────────────────────┐
                │        Docs/        │
                │  iterations/ (活跃)  │
                │  archive/    (归档)  │
                │  project/    (现状)  │
                └──────────┬──────────┘
                           │ 归档时触发
                           ▼
                    ┌──────────────┐
                    │ AI-CONTEXT.md│
                    │  (摘要+索引)  │
                    └──────────────┘
```
