# Docs/ 目录结构规范

> **文档定位**：定义 Docs/ 目录的完整结构和每个目录/文件的职责。
>
> **创建日期**：2026-06-19

---

## 1. 完整目录树

```
Docs/
├── AI-CONTEXT.md                        # AI 工作记忆（摘要 + 索引）
├── .ai-context-sync.json                # AI Context 同步元数据（Skill 内部用）
│
├── project/                             # 📌 项目当前静态状态
│   ├── architecture.md                  #   系统架构全景
│   ├── data-model.md                    #   数据模型（当前 ER 关系 + 表结构概览）
│   ├── api-contracts.md                 #   接口总览（当前有效接口）
│   └── modules/                         #   按模块拆分的业务现状
│       ├── {module-a}.md
│       ├── {module-b}.md
│       └── ...
│
├── iterations/                          # 🔄 活跃迭代（进行中 + 已完成待归档）
│   └── {YYYY-MM-DD}_{需求名}_{版本}/
│       ├── _meta.yaml                   #   迭代元信息
│       ├── prd.md                       #   产品需求文档
│       ├── tech-design.md               #   技术设计文档
│       ├── tasks.md                     #   任务分解与状态
│       ├── ddl-changes.md               #   DDL 变更记录
│       ├── api-changes.md               #   接口变更记录
│       └── review-notes.md              #   评审与决策记录
│
└── archive/                             # 📦 已归档迭代
    └── {YYYY-MM-DD}_{需求名}_{版本}/
        └── ...（结构同 iterations/，只读）
```

---

## 2. 各目录职责

### 2.1 根级文件

| 文件 | 维护者 | 更新时机 | 职责 |
|------|--------|---------|------|
| `AI-CONTEXT.md` | 同步 Skill（AI 维护） | 归档时 / 手动触发 / 定时 | Agent 日常工作记忆，Docs 的摘要 + 索引 |
| `.ai-context-sync.json` | 同步 Skill（AI 维护） | 每次同步时 | 同步快照（文件 hash + mtime），用于增量更新 |

### 2.2 project/ — 项目当前状态

**核心约束**：只在迭代归档时更新，迭代进行中保持不变。

| 文件 | 职责 | 更新来源 |
|------|------|---------|
| `architecture.md` | 系统架构全景：模块关系图、技术选型、部署拓扑、核心约束 | 归档时合并 `tech-design.md`（如有架构变更） |
| `data-model.md` | 当前有效 ER 图、全量表结构概览、表间关系 | 归档时合并 `ddl-changes.md` |
| `api-contracts.md` | 当前有效接口总览：路径、方法、核心字段、状态 | 归档时合并 `api-changes.md` |
| `modules/*.md` | 单模块业务现状：业务规则、核心流程、边界条件 | 归档时合并 `prd.md` + `tech-design.md` |

**命名规范**：
- `modules/` 下的文件名 = 模块标识符（如 `farvis-video.md`、`farvis-payment.md`）
- 模块标识符与代码中的模块目录名保持一致

### 2.3 iterations/ — 活跃迭代

**核心约束**：一个迭代 = 一个目录 = 全部资料集中存放。

| 文件 | 职责 | 说明 |
|------|------|------|
| `_meta.yaml` | 迭代元信息 | 状态、时间、影响模块、摘要（详见迭代模板） |
| `prd.md` | 产品需求文档 | 本次迭代的产品需求定义 |
| `tech-design.md` | 技术设计文档 | 技术方案、架构决策 |
| `tasks.md` | 任务分解与状态 | 开发任务列表、进度跟踪 |
| `ddl-changes.md` | DDL 变更记录 | 本次迭代涉及的表结构变更 |
| `api-changes.md` | 接口变更记录 | 新增/修改/废弃的接口 |
| `review-notes.md` | 评审与决策记录 | 关键决策、评审意见、取舍记录 |

**目录命名规范**：`{YYYY-MM-DD}_{需求名称}_{版本号}`
- 日期：迭代启动日期，保证时间排序
- 需求名称：简短中文或英文，描述核心需求
- 版本号：v1.0 / v1.1 等，标记迭代轮次
- 示例：`2026-06-19_用户认证_v1.0`、`2026-07-01_payment-integration_v2.0`

### 2.4 archive/ — 已归档迭代

结构与 `iterations/` 完全一致，区别：
- 只读，归档后不再修改
- 由归档流程从 `iterations/` 迁移而来

## 3. 与 Git 分支的对应关系

```
Git 分支                          Docs 迭代目录
────────────────────────────────────────────────────
main                              project/（项目现状）
2026-06-19_用户认证_v1.0    ↔     iterations/2026-06-19_用户认证_v1.0/
(合并到 main 后)                  archive/2026-06-19_用户认证_v1.0/
```

**分支命名 = 迭代目录名**，生命周期一一对应。

## 4. 文件所有权

| 区域 | 主要维护者 | 协作方式 |
|------|-----------|---------|
| `AI-CONTEXT.md` | AI（同步 Skill） | AI 自动维护，人可审阅 |
| `.ai-context-sync.json` | AI（同步 Skill） | 纯机器维护，人不需要关注 |
| `project/*` | 人机协同 | 归档时 AI 合并，人审阅确认 |
| `iterations/*` | 人机协同 | 迭代中人和 AI 共同编写 |
| `archive/*` | 无（只读） | 归档后冻结 |

---

## 5. 设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| project/ 更新时机 | 仅归档时 | 保证现状文档的稳定性，迭代中的变更不影响"真相源" |
| 迭代目录集中 vs 分散 | 集中存放 | 找资料不用满世界翻，归档也简单（整个目录 mv） |
| archive/ vs 留在 iterations/ | 独立目录 | 语义清晰，一眼看出哪些是活跃的、哪些是历史的 |
| modules/ 文件粒度 | 一个模块一个文件 | 粒度适中——太粗找不到细节，太细管理成本高 |
| .ai-context-sync.json 位置 | Docs 根级 | 紧跟 AI-CONTEXT.md，但用 `.` 前缀表示是内部文件 |
