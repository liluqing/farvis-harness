# 人机协作边界

> **何时加载**：Agent 不确定是否该继续自主推进，或需要判断是否必须问用户时。

---

## 核心原则

让 Agent 尽量自己往下跑，不要动不动停下来问人。

人负责目标和关键决策，Agent 负责执行、验证、修复和文档维护。

---

## 人必须介入

| 场景 | 原因 |
|------|------|
| 定义迭代目标 | 产品方向不能由 Agent 替人决定 |
| 影响交付范围 | 这期做不做、优先级如何，需要人定 |
| 影响业务规则 | 计费、审批、权限、合规等需要业务判断 |
| 影响架构选型且没有上下文依据 | 需要团队技术取舍 |
| 同一问题自修 3 次仍失败 | 已进入真实卡点 |
| 设计问题阻塞交付 | 需要人判断方向 |
| 最终验收 | 是否交付由人决定 |

---

## Agent 应该自主推进

| 场景 | 做法 |
|------|------|
| 行业通用做法 | 直接定，摘要中说明 |
| 项目已有同类模式 | 复用已有模式 |
| 小范围实现决策 | 自定并记录 |
| 测试失败但根因清楚 | 自己分析、修复、重跑 |
| 文档路径明确 | 自己更新文档和 state |
| 切片完成且验证通过 | 直接进入下一切片 |

---

## 灰色决策

既不适合打断用户，又不适合完全忽略时：

1. 采用最通用、最保守的默认做法
2. 标注 `[待确认]`
3. 在阶段概要或交付报告里集中列出

---

## 分支规则

| 分支 | 触发条件 | 动作 |
|------|----------|------|
| A：Agent 自定 | 上下文有约定、行业标准、已有参考、影响小 | 直接决定并记录 |
| B：必须问用户 | 影响范围、业务、架构、外部系统、合规 | 给 2~3 个选项 + 建议 |
| C：暂定待确认 | 不满足 A/B，但可临时推进 | 默认方案 + `[待确认]` |

Phase 1 的分支 B 每轮最多 5 个问题。
Phase 3 的分支 B 每轮最多 3 个问题。
