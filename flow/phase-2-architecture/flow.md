# Phase 2：架构设计与任务规划

> **定位：** 全局架构设计 + 粗粒度任务拆解。回答「怎么做」，是 Phase 1（做什么）和 Phase 3（做出来）之间的桥梁。
> **执行角色：** 架构师
> **输入：** Phase 1 的 PRD + 原型
> **产出：** 架构文档 + 任务清单（粗粒度）
> **适用：** 多模块项目 / 涉及新系统设计 / 涉及技术选型 / 用户明确要求

---

## 何时走 Phase 2

| 条件 | 走不走 | 说明 |
|------|:--:|------|
| 业务模块 ≥ 3 个独立域 | ✅ 走 | 如：用户、订单、支付、库存 |
| 涉及技术选型 | ✅ 走 | 框架/数据库/中间件需要选择 |
| 新系统从零搭建 | ✅ 走 | 无历史架构可复用 |
| 多服务/微服务架构 | ✅ 走 | 需要明确服务边界和通信方式 |
| 简单 CRUD 单模块 | ❌ 跳过 | Phase 3 自带技术设计 |
| 已有同类模块可参考 | ❌ 跳过 | 复用已有架构模式 |
| 用户说「不用设计架构，直接开发」 | ❌ 跳过 | 听用户的 |

> **Agent 已定：** 是否走 Phase 2 由 Agent 根据以上条件自动判定，在 Phase 1 完成后告知用户「建议走 Phase 2」或「可直接进入 Phase 3」，让用户确认。

---

## 启动检查

Phase 2 启动时，Agent 先做三件事：

### 0. Harness 预检（Java 项目）

如果项目根目录存在 `.harness/`，**优先加载**：

```
1. 读 .harness/ai-context/project-map.yaml    → 已有模块划分（如已初始化）
2. 读 .harness/ai-context/business-rules.yaml → 已有业务约束
3. 读 .harness/ai-context/coding-rules.yaml   → 已有编码规则
4. 读 .harness/ai-context/error-catalog.yaml  → 已有错误码
```

如果这些文件为空模板（刚初始化）→ Phase 2 的职责就是填它们。

### 1. 确认输入

- [ ] Phase 1 PRD 已就绪（`docs/product/prd-*.md`）
- [ ] 原型文件（如有）可访问
- [ ] 项目上下文（AGENTS.md / README）已读取

### 2. 确认产出目标

Phase 2 的使命：把 PRD 的「做什么」翻译成「怎么做、谁做、做多少」。

**产出清单**（项目已接入 `.harness/` 时同时写入）：

| 产出 | 写入位置 | Harness 同步 |
|------|---------|-------------|
| 模块划分 | `docs/architecture/architecture-*.md` | `.harness/ai-context/project-map.yaml` |
| ADR | `docs/architecture/architecture-*.md`（嵌入） | `.harness/ai-context/adr/ADR-*.md`（独立文件） |
| 编码约束更新 | — | `.harness/ai-context/coding-rules.yaml`（如模块划分引入新约束） |
| 任务清单 | `docs/architecture/task-list-*.md` | — |

## 流程总览

```
PRD + 项目上下文
    ↓
STEP 1: 技术选型
    ↓
STEP 2: 模块划分
    ↓
STEP 3: 架构决策记录（ADR）
    ↓
STEP 4: 粗粒度任务拆解
    ↓
STEP 5: 任务依赖 + 排序 → 用户确认
    ↓
产出：架构文档 + 任务清单
```

## Phase 2 vs Phase 3 职责边界

这是全链路最关键的分工：

| 维度 | Phase 2（全局） | Phase 3（单任务） |
|------|------|------|
| **粒度** | **粗**——按业务模块拆 | **细**——按端点/页面/命令拆 |
| **产物示例** | 「用户管理模块」「订单处理模块」 | 「POST /api/v1/users」「用户列表页」 |
| **数量关系** | 1 个 Phase 2 任务 | → N 个 Phase 3 切片 |
| **技术设计** | 技术选型、模块划分、通信方式、部署架构 | 单端点 API 契约、Entity、DB 表 |
| **决策层级** | 架构决策（影响全局） | 实现决策（影响本切片） |
| **变更成本** | 高——改了架构所有任务受影响 | 低——只影响本切片 |

## STEP 1：技术选型

基于 PRD 的功能范围和非功能需求，确定技术栈：

| 维度 | 决策项 | 依据 |
|------|------|------|
| **语言/运行时** | <!-- 如 Java 17, Python 3.12, Go 1.22 --> | 团队能力 / 项目上下文 / 性能需求 |
| **框架** | <!-- 如 Spring Boot 3.x, FastAPI, Gin --> | 生态成熟度 / 项目规模 |
| **数据库** | <!-- 如 MySQL 8.0, PostgreSQL 15, MongoDB --> | 数据结构 / 一致性需求 / 查询模式 |
| **缓存** | <!-- 如 Redis, Caffeine, 无 --> | 读写比例 / 实时性要求 |
| **消息队列** | <!-- 如 RabbitMQ, Kafka, 无 --> | 异步需求 / 吞吐量 |
| **部署** | <!-- 如 Docker Compose, K8s, 云服务 --> | 运维能力 / 规模 |

> **决策规则：**
> - 项目上下文中已有约定的 → 直接引用，不重新选
> - 团队有明确技术栈偏好的 → 尊重偏好
> - 首次选型 → 给 2 个主流选项 + 建议

## STEP 2：模块划分

### 2.1 模块识别

从 PRD 的功能范围中识别业务模块：

```
功能 A ─┐
功能 B ─┤ → 模块 1：<!-- 模块名 -->
功能 C ─┘

功能 D ─┐
功能 E ─┤ → 模块 2：<!-- 模块名 -->
```

### 2.2 模块定义

| 模块 | 职责 | 核心实体 | 对外接口 |
|------|------|------|------|
| <!-- 模块 1 --> | <!-- 一句话职责 --> | <!-- Entity 列表 --> | <!-- 暴露哪些 API/事件 --> |
| <!-- 模块 2 --> | <!-- 一句话职责 --> | <!-- Entity 列表 --> | <!-- 暴露哪些 API/事件 --> |

### 2.3 模块间通信

```
模块 1 ──REST──▶ 模块 2
模块 1 ──事件──▶ 模块 3
模块 2 ──同步调用──▶ 模块 3
```

### 2.4 同步到 Harness（如 `.harness/` 存在）

模块划分完成后**立即**写入 `.harness/ai-context/project-map.yaml`：

```yaml
# 自动生成，勿手动修改
project:
  name: "<来自 Phase 1 PRD 的项目名>"
  architecture: "<layered-monolith / modular-monolith / microservices>"
  language: "<java-17 / java-21>"
  framework: "<spring-boot-3.x>"

modules:
  - name: "<模块 1>"
    responsibility: "<一句话职责>"
    entities: ["<Entity>", ...]
    interfaces: ["<API/事件>", ...]
    depends_on: ["<模块 2>"]
  - name: "<模块 2>"
    ...
```

> 写入后提示用户：「模块划分已同步到 `.harness/ai-context/project-map.yaml`」

## STEP 4：粗粒度任务拆解

### 4.1 拆解原则

- 一个任务 = 一个可独立交付的业务模块
- 一个任务 = N 个 Phase 3 切片（通常 3~8 个端点/页面）
- 任务之间有依赖的标出来，无依赖的可并行

### 4.2 任务清单

| 编号 | 任务名 | 所属模块 | 涉及实体 | 预估切片数 | 依赖 | 优先级 |
|:--|------|------|------|:--:|------|:--:|
| T1 | <!-- 任务名 --> | <!-- 模块 --> | <!-- Entity --> | ~N | 无 | P0 |
| T2 | <!-- 任务名 --> | <!-- 模块 --> | <!-- Entity --> | ~N | T1 | P0 |
| T3 | <!-- 任务名 --> | <!-- 模块 --> | <!-- Entity --> | ~N | T1, T2 | P1 |

### 4.3 每个任务的内容概要

## STEP 5：任务依赖 + 排序 → 用户确认

### 5.1 依赖图

```
T1（用户模块）
  ├──▶ T2（订单模块）──▶ T4（支付模块）
  └──▶ T3（商品模块）
```

### 5.2 建议执行顺序

| 顺序 | 任务 | 理由 |
|:--:|------|------|
| 1 | T1 | 无依赖，基础模块 |
| 2 | T2, T3 | 依赖 T1，可并行 |
| 3 | T4 | 依赖 T2 |

### 5.3 概要确认

输出架构概要给用户确认：

```markdown
【架构概要】

**技术栈：** <语言> + <框架> + <数据库>

**模块划分：** N 个模块 — <模块 1>、<模块 2>、<模块 3>

**任务拆解：** M 个任务 — <T1>、<T2>、<T3>...

**关键决策：**
- <ADR-1 标题>：选了 <X>，因为 <Y>
- <ADR-2 标题>：选了 <X>，因为 <Y>

**需要你定（如有）：**
1. <技术选型分歧> → A/B + 建议
```

用户确认后 → 展开完整架构文档。

```

- 说「开始开发」→ 进入 Phase 3（Spec & 开发）
```
