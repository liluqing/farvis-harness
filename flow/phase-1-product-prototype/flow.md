# Phase 1：产品与原型

> **定位：** 将用户原始需求转化为 PRD + 轻量原型。是整个 Harness 流程的起点。
> **Agent 帽子：** 产品经理 + 交互设计师
> **输入：** 用户原始需求（聊天记录、想法、竞品链接、文档）
> **产出：** `prd/<功能名>-prd.md` + 可选原型 HTML
> **适用：** 通用——B 端后台、C 端 App、小程序、Web 页面、API 服务、工具型产品等

---

## 流程总览

```
用户原始需求
    ↓
STEP 0: 产品形态识别（新增）
    ↓
STEP 1: 收集与整理
    ↓
STEP 2: 需求澄清（产品层决策树，含灰色地带处理）
    ↓
STEP 3: PRD 概要 → 用户确认
    ↓
STEP 4: 生成轻量原型（可选，带硬约束 + 自检）
    ↓
STEP 5: 用户反馈 → 调整（有终止条件）
    ↓
STEP 6: 阶段退出 → Done Checklist + Phase 3 交接
```

---

## STEP 0：产品形态识别

Agent 先判断产品形态，这影响后续的模板倾向和决策示例。

| 形态 | 典型特征 | 原型侧重 |
|------|---------|---------|
| **B 端管理后台** | 列表/表单/数据看板、角色权限、审批流 | 页面间数据流转、权限视角切换 |
| **C 端 App/H5** | 注册登录、内容消费、社交互动、支付 | 核心操作路径、关键状态变化 |
| **小程序** | 轻量入口、微信生态、扫码/分享 | 快速操作流程、授权交互 |
| **营销落地页** | 单页或少页、转化漏斗、表单收集 | 页面滚动叙事、CTA 位置 |
| **API/后端服务** | 无 UI、接口定义、数据模型 | 不出原型，用 API 文档替代 |
| **工具/CLI 产品** | 命令行或单功能工具 | 不出原型，用示例命令替代 |

> **Agent 已定：** 形态判断基于用户描述关键词（如提到"后台""管理"→B 端；"App""小程序"→对应形态）。如果无法从描述中判定，走分支 C 暂定"Web 应用"并在概要中标注。

## STEP 1：收集与整理

Agent 读取用户提供的所有信息，产出内部「需求快照」：

```
【需求快照】<功能名>
形态：<STEP 0 判定结果>

一、已知信息
- 用户明确提到的需求点
- 用户提到的场景/角色
- 用户给的参考/竞品

二、信息缺口
- 需要澄清的业务规则
- 需要确认的范围边界
- 需要明确的优先级
```

不发给用户——Agent 内部整理，用于准备 STEP 2 的问题。

## STEP 2：需求澄清（产品层决策树）

### 2.1 三条分支

| 分支 | 触发条件 | 处理方式 |
|:----|------|------|
| **分支 A：Agent 自定** | 行业通用模式 / 竞品标准做法 / 对方向影响小 / 项目已有同类参考 | 直接定，PRD 概要中一句告知 |
| **分支 B：必须问用户** | 影响业务逻辑 / 影响交付范围 / 影响产品定位 / 涉及合规法务 / 多部门利益冲突 | 给 2~3 选项 + 产品建议 |
| **分支 C：暂定 + 标记** | 既不满足 A 也不满足 B / 可临时用默认做法顶替 | 用行业默认，标注 `[待确认]`，PRD 概要中一并确认 |

### 2.2 分支 C 的设计意图

决策树的灰度区：Agent 觉得"不太对直接拍板，但也不至于为这事打断用户"。比如"消息推送用站内信还是短信"——重要但可后置。

处理规则：
- 选最通用/最保守的做法暂定
- 标注 `[待确认]`，注明选了谁和为什么
- 统一在 PRD 概要的"暂定待确认"区列出
- 用户确认概要时一次性处理所有灰色决策

### 2.3 分支判断示例（通用版）

**分支 A（Agent 自定）：**

| 条件 | 通用示例 |
|------|------|
| 行业通用产品模式 | 用户注册流程、搜索+筛选、分页加载 |
| 竞品有标准做法 | Tab 切换、下拉刷新、滑动删除 |
| 对功能方向影响小 | 默认排序规则、按钮文案、占位文案 |
| 项目已有同类可参考 | 已有的模块模式、已确立的交互规范 |

Agent 自定时在 PRD 概要中一句告知：
> **Agent 已定：** 列表默认按创建时间倒序；每页 20 条；搜索支持模糊匹配。

**分支 B（必须问用户）：**

| 条件 | 通用示例 |
|------|------|
| 影响业务逻辑 | 审批规则、计费方式、推荐算法策略 |
| 影响交付范围 | 这期做不做 iOS、做不做数据导出 |
| 影响产品定位 | 面向内部人员还是外部客户、免费还是付费 |
| 涉及合规/法务 | 数据留存期限、隐私合规、金融合规 |
| 多部门利益冲突 | A 部门需要审核，B 部门要求自助 |

走分支 B 时给 2~3 选项 + 建议：
> **需要你定：**
> 1. 用户注册方式 → A. 手机号+验证码 B. 邮箱注册 C. 手机号+微信授权。建议 A，覆盖最广。

**分支 C（暂定 + 标记）：**

| 场景 | 暂定做法 | 标注示例 |
|------|---------|------|
| 通知方式不确定 | 站内信（最通用） | `[待确认] 消息通知暂用站内信，后续可按需加短信/推送` |
| 数据范围待定 | 全部显示（先不做权限） | `[待确认] 暂不引入角色权限，所有用户看到相同数据` |
| 支付对接方案未定 | 留接口不实现 | `[待确认] 支付流程界面保留，后端对接方案待定` |

### 2.4 产品层约束

- 分支 B 问题每轮 ≤ 5 个（超过 → 优先问范围/定位问题，其余走分支 C）
- 分支 C 不限数量，但每个都要有"暂定选了什么 + 为什么"
- 如果所有问题都是分支 B 且 > 5 个 → 优先问范围/业务逻辑/合规，交互细节走分支 C

### 2.5 产品层 vs Spec 层对比

| 维度 | 产品层（Phase 1） | Spec 层（Phase 3） |
|------|------------------|---------------------|
| Agent 定的范围 | 行业产品模式、竞品做法 | 技术约定、命名规范 |
| 必须问的范围 | 业务逻辑、范围、定位、合规 | 数据策略、架构选择 |
| 灰色地带 | 暂定 + 标记 | 暂定 + 标记 |
| 分支 B 上限 | 5 个 | 3 个 |

## STEP 3：PRD 概要 → 用户确认

> 概要模板见 `templates/prd-summary.md`。核心原则：**先让业务方看懂「产品骨架」（流程+状态+实体），再谈功能清单。**

### 概要包含

| 章节 | 说明 | 为什么需要 |
|------|------|------|
| **产品三问** | 痛点、用户、成功标准 | 业务方一眼看方向 |
| **核心流程** | 用户完成核心任务的端到端路径，2~3 个 | 确认操作方式对不对 |
| **核心状态流转** | 关键实体的状态机：有哪些状态、什么条件触发变化 | Phase 3 开发时的状态机依据 |
| **核心实体** | 产品有哪些关键数据对象，以及它们的关系 | 开发前对齐数据模型骨架 |
| **功能范围** | 本期/后续 | 确认交付边界 |
| **关键业务规则** | 对开发有影响的业务逻辑 | 防止开发理解偏差 |
| **决策汇总** | Agent 已定 / 暂定待确认 / 需要你定 | 三类决策一个不落 |

发给用户 → **等确认** → 通过后展开完整 PRD（模板：`templates/prd.md`）。

## STEP 4：生成轻量原型（可选）

### 4.1 触发条件

满足以下任一即建议出原型（API/CLI 类产品除外）：

| 条件 | 说明 |
|------|------|
| ≥ 3 个页面间数据流转 | 业务方需要看到数据怎么在页面间流动 |
| 业务规则需在界面体现 | 如状态流转、权限差异、条件分支 |
| 用户说「先看看效果」 | 业务方需要视觉确认 |
| 多角色视角 | 不同角色看到不同界面/数据 |
| 核心流程含关键决策点 | 如支付、提交、确认等不可逆操作 |

### 4.2 原型分层目的

| 优先级 | 目的 | 服务对象 | 设计要求 |
|:--:|------|------|------|
| **第一** | 对齐需求 | 业务方 | 数据真实、核心流程完整、状态变化可感知 |
| **第二** | 对齐认知 | 开发者 | 交互逻辑正确、页面跳转准确、异常态覆盖 |

### 4.3 硬约束

| 约束项 | 上限 | 说明 |
|------|:--:|------|
| 页面数 | ≤ 5 | 核心页面即可，不覆盖设置/个人中心等边缘页面 |
| 弹窗/浮层 | ≤ 3 | 只做关键弹窗（确认、表单提交、详情） |
| CSS 行数 | ≤ 800 | 不做视觉设计，CSS 只用于布局和基础美观 |
| 外部依赖 | 0 | 不引外部 CSS/JS/CDN |
| mock API | 0 | 纯静态数据，不写 fetch/axios |

### 4.4 原型产出标准

一个自包含 HTML 文件，必须满足：

**给业务方验收：**
- [ ] 核心操作路径完整（从头走到尾不卡住）
- [ ] 数据真实感（不用「测试数据」「XXX」「111」）
- [ ] 关键状态可感知（加载中/空数据/错误提示/成功反馈）
- [ ] 页面跳转逻辑正确
- [ ] 不可逆操作有确认（删除、提交、支付）

**给开发者对齐：**
- [ ] 交互逻辑与 PRD 一致
- [ ] 各页面状态覆盖（正常态/空态/错误态/边界态）
- [ ] 多角色视角覆盖（如适用）
- [ ] 页面间数据流转逻辑清晰

### 4.5 原型流程

```
PRD 概要确认
    ↓
Agent 生成原型简报（模板：templates/prototype-brief.md）
    ↓
Agent 生成原型 HTML
    ↓
Agent 自检（对照产出标准逐项打勾）
    ↓
交付用户 → 连同 PRD 一起展示
```

## STEP 5：用户反馈 → 调整

用户对 PRD 和原型提出反馈 → Agent 调整 → 再次确认。

### 终止条件

| 条件 | 处理 |
|------|------|
| 用户明确说「可以进入开发」「没问题」「就这样」 | ✅ 正常退出，进入 STEP 6 |
| 循环 ≥ 3 轮且无实质性新反馈 | Agent 主动提示：「当前 PRD 已迭代 3 轮，是否先冻结这版，边开发边微调？」 |
| 用户说「先这样，后面再说」 | ✅ 冻结 PRD，未解决的 `[待确认]` 标记为 「已冻结，Phase 3 前需确认」 |

### 冻结标记

当 PRD 以"冻结"方式退出时，在 PRD 头部标注：

```
> ⚠️ 状态：已冻结（Phase 3 启动前需确认 X 个遗留项）
```

## STEP 6：阶段退出与交接

### 6.1 Done Checklist（Agent 自检）

完成以下所有项才算 Phase 1 完成：

- [ ] **PRD 已写入** `prd/<功能名>-prd.md`
- [ ] 所有分支 B 问题已由用户确认
- [ ] 所有分支 C `[待确认]` 已处理（确认 or 冻结）
- [ ] 产品三问有明确答案
- [ ] 成功标准可衡量
- [ ] 功能范围本期/后续边界清晰
- [ ] 关键业务规则明确
- [ ] 非功能需求有初步识别（性能/安全/合规至少各提一句，无则写"无特殊要求"）
- [ ] 原型（如有）已自检通过
- [ ] 用户确认了最终 PRD

### 6.2 Phase 3 交接清单

在 PRD 末尾写入以下章节，供 Phase 3 Agent 读取：

```markdown
## Phase 3 启动条件

> **Phase 1 完成时间：** YYYY-MM-DD
> **Phase 1 执行者：** Agent (harness-dev-flow)

### 状态
- [ ] 所有 [待确认] 已解决或冻结
- [ ] 功能可按独立切片交付
- [ ] 技术栈已明确或标记为「待 Phase 2 确定」

### 遗留项（如有）
- [冻结] XXX — 冻结原因，建议何时确认

### 原型（如有）
- 文件：<路径>
- 已覆盖页面：<列表>

### 已知风险
- 风险 1：<描述>
- 风险 2：<描述>
```

### 6.3 Phase 1 完成

Agent 输出：

> **Phase 1 完成。** PRD 已就绪，原型已交付。可以进入 Phase 2（架构设计）或 Phase 3（Spec & 开发）。说「开发 XX 功能」即可启动。

### 6.4 Harness 上下文交出（如项目已接入 Java Harness）

如果项目根目录存在 `.harness/`，Phase 1 完成后**自动提取** PRD 中的业务约束到 business-rules.yaml。

#### 6.4.1 提取来源

扫描 PRD 以下章节：

| PRD 章节 | 提取什么 | business-rules.yaml 键 |
|---------|---------|----------------------|
| 功能范围 | 每个功能的操作类型 | `<operation>:`（如 `order-creation:`） |
| 业务规则 | 幂等要求、唯一约束 | `idempotency-key`, `unique-index` |
| 非功能需求 | 性能/安全/合规/可靠性 | `consistency`, `timeout`, `isolation` |
| 数据流 | 读写路径、缓存策略 | `read-path`, `cache-policy`, `cache-ttl` |
| 访问控制 | 权限约束 | `access-control` |
| 外部集成 | 外部 API 调用 | `circuit-breaker`, `rate-limit`, `retry-policy` |
| 错误处理 | 业务异常场景 | `error-scenarios` |

#### 6.4.2 提取示例

假设 PRD 有如下描述：

> 「用户下单时需要保证不重复扣款，每个 requestId 只能处理一次。下单成功后通过消息队列通知仓储。扣减库存时使用乐观锁防止超卖。」

Agent 提取为：

```yaml
# .harness/ai-context/business-rules.yaml
rules:
  order-creation:
    idempotency-key: "requestId"
    unique-index: "uk_order_request_id"
    consistency: "db-transaction + outbox"
    cache-policy: "no-cache"
    error-scenarios:
      - "DUPLICATE_REQUEST → 返回已有订单"
      - "INSUFFICIENT_STOCK → 返回库存不足错误码"

  stock-deduction:
    isolation: "optimistic-lock"
    retry-policy: "3 times with exponential backoff"
    error-scenarios:
      - "VERSION_CONFLICT → 自动重试，超过 3 次返回失败"
```

#### 6.4.3 提取规则

1. **每条规则必须可验证** — 「高性能」不写，写成「P99 < 200ms」
2. **幂等键必须明确到字段** — 不写「需要幂等」，写 `idempotency-key: "requestId"`
3. **外部 API 必须有三件套** — 超时 + 重试 + 熔断参数
4. **未知的填占位** — 不确定的写 `<!-- 待 Phase 2 确定 -->`
5. **每个错误场景对应 error-catalog 一条** — 同步写入骨架

#### 6.4.4 执行

```
1. 读 PRD → 按 6.4.1 表逐章扫描
2. 生成 business-rules.yaml 内容
3. 写入 .harness/ai-context/business-rules.yaml
4. 输出摘要：「从 PRD 提取了 X 条业务约束、Y 个错误场景」
5. 对用户说：「已提取到 .harness/ai-context/business-rules.yaml，请确认。也可以说『开始 Phase 2』进入架构设计。」
```

如果用户拒绝或跳过 → 标注 `[待提取]`，Phase 2 时再处理。

#### 6.4.5 同步写入 error-catalog.yaml

从 PRD 中识别到的每个业务异常场景，**同时**写入 error-catalog.yaml：

**提取来源：**

| PRD 内容 | error-catalog 字段 |
|---------|-------------------|
| 「XX 情况返回 YY 错误」 | `cause` + `action` |
| 「XX 操作失败时提示 YY」 | `http_status` 推断（400/402/409/503） |
| 「XX 需要重试」 | `action` 含重试策略 |

**提取示例：**

PRD 描述：「重复下单返回已有订单，不扣款」「库存不足返回错误码提示加购失败」「支付超时 15 分钟自动取消」

Agent 同步写入：

```yaml
# .harness/ai-context/error-catalog.yaml
errors:
  DUPLICATE_REQUEST:
    cause: "重复 requestId 导致幂等命中"
    action: "返回已有订单结果，不重复处理"
    http_status: 200
    log_level: "INFO"

  INSUFFICIENT_STOCK:
    cause: "库存不足，无法满足订单数量"
    action: "返回错误码 + 引导用户修改数量或等待补货"
    http_status: 409
    log_level: "WARN"

  PAYMENT_TIMEOUT:
    cause: "支付超过 15 分钟未完成"
    action: "自动取消订单 + 释放库存 + Outbox 通知"
    http_status: 408
    log_level: "WARN"
```

**关键规则：**
1. 错误码命名用大写蛇形（`DUPLICATE_REQUEST`），与 BizException 的 code 参数一致
2. `http_status` 用语义正确的状态码（400 参数错、409 冲突、503 下游超时）
3. 每个错误场景一条记录，不合并不同原因的错误

执行完 6.4.4 之后 → 读 business-rules.yaml 中的 `error-scenarios` → 转写为 error-catalog.yaml → 输出：「同步写入了 Z 个错误码到 error-catalog.yaml」。
