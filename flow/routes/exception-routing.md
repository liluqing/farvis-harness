# 异常路由

> **何时加载**：用户不按标准“新需求 → Phase 1”路径走时。

---

## 路由表

| 用户输入 | 问题 | 处理 |
|----------|------|------|
| “照着这个做” + 原型/截图/竞品 | 缺少文字 PRD | 从原型反推 PRD |
| “改一下 XX” | 已有功能变更 | 变更模式 |
| “把 JPA 换成 MyBatis” | 技术重构 | 技术重构模式 |
| “Spring Boot 怎么配置多数据源” | 技术咨询 | 不进入 Phase，直接回答 |
| “做一个电商平台” | 需求过大 | 拆迭代规划 |
| “写个登录功能”但无 PRD | 想跳过产品定义 | 先评估规模，小型走计划，中型以上进 Phase 1 |

---

## From Prototype

当用户给的是原型、截图、竞品链接，而不是文字需求：

```
用户提供原型
  → Agent 分析页面、功能、交互、角色、状态
  → 提取功能点和业务规则
  → 产出 PRD 概要
  → 标注 [推断] 和 [待确认]
  → 用户确认/修正
  → 进入 Phase 2 或 Phase 3
```

关键：原型不是完整需求。Agent 要补业务规则、异常处理、边界条件。

## Change Mode

当用户修改已有功能：

```
用户描述变更
  → 读取相关历史文档
      - Docs/project/modules/{module}.md
      - Docs/archive/*/prd.md
      - Docs/archive/*/tech-design.md
      - .harness/ai-context/project-map.yaml
  → 找到对应代码
  → 做跨迭代影响分析
  → 评估规模
      - 微型：直接改
      - 小型：计划模式
      - 中型+：增量 PRD + 标准流程
  → 执行变更
  → 同步文档和 ai-context
```

变更后必须更新文档，否则文档和代码会脱节。

## Technical Refactor

当用户要求技术重构：

```
用户描述重构目标
  → Agent 评估当前方案和目标方案
  → 识别影响范围、风险、回滚方案
  → 输出重构计划
  → 用户确认
  → 分切片执行
  → 每个切片验证兼容性
```

重构计划必须包含：

- 范围
- 分阶段策略
- 回滚方案
- 数据迁移策略（如有）
- 测试策略
- 风险点

---

## Non Development

如果用户只是问技术问题、解释概念、查询项目状态：

- 不进入 Phase
- 不修改 state.json
- 不创建迭代目录
- 直接回答或读取相关文档后回答

如果回答中发现用户其实要落地修改，再回到 `routes/demand-sizing.md`。

## 跨迭代影响分析

触发：新需求涉及已有模块修改。

步骤：

1. 读取模块文档：`Docs/project/modules/{module}.md`
2. 读取结构化上下文：`.harness/ai-context/project-map.yaml`
3. 查找依赖方、调用方、数据表、外部接口
4. 输出影响分析：

```markdown
## 影响分析：<变更内容>

### 直接影响
- <模块>：<改动>

### 间接影响
- <依赖方>：<影响>

### 风险评估
| 风险 | 影响 | 概率 | 缓解 |
|------|------|------|------|

### 建议
- 修改顺序：
- 测试重点：
```

影响范围大于 3 个模块时，建议拆成多个迭代。
