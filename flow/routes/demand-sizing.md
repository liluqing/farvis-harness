# 需求规模评估与四级路由

> **何时加载**：用户提出新需求、变更、修复或重构请求时。

---

## 目标

先判断规模，再决定流程。不要无脑进入 Phase 1。

---

## 四级分类

| 级别 | 判断标准 | 流程 | 人工介入 |
|------|----------|------|----------|
| 微型 | ≤ 2 个文件变更，≤ 30 行代码 | 直接改，不走 Phase | 无 |
| 小型 | 3+ 文件或 30+ 行，但 ≤ 3 个切片 | 计划模式 | 确认计划 |
| 中型 | 4~10 个切片 | 标准 Phase 流程 | 目标、拆解、验收 |
| 大型 | >10 个切片，或跨多个领域/模块 | 先拆迭代 | 拆迭代规划 |

---

## 判断维度

1. 涉及几个模块？
2. 预计修改多少文件？
3. 预计新增多少文件？
4. 是否有跨模块接口变更？
5. 是否涉及数据模型变更？
6. 是否引入新技术组件？
7. 是否涉及支付、权限、合规、外部系统等横切关注点？

---

## 微型变更

适用：配置、文案、typo、单字段、单个小 bug。

流程：

```
用户描述变更
  → Agent 判断为微型
  → 直接改
  → 跑相关测试或静态检查
  → 更新 state.json（如有活跃迭代则记录为 ad-hoc）
  → 报告完成
```

不产出 PRD、架构文档、技术设计。

---

## 小型需求：计划模式

适用：一个简单 CRUD、一个配置项及逻辑、一个小页面、一个小接口组。

流程：

```
用户描述需求
  → Agent 输出变更计划
  → 用户确认
  → Agent 分步骤实施
  → 每步自验
  → 更新相关文档和 state
  → 报告完成
```

计划模板：

```markdown
## 变更计划：<功能名>

### 需求理解
<一句话描述>

### 影响范围
- <文件/模块>：<修改说明>

### 实施步骤
1. <步骤>
2. <步骤>

### 测试策略
- 单元测试：
- 集成测试：

### 风险点
- <风险，如无写“无明显风险”>
```

---

## 中型需求：标准 Phase 流程

适用：完整模块、多接口、前后端组合、涉及数据模型。

加载：

1. `routes/phase-routing.md`
2. `protocols/doc-health-check.md`
3. 命中的 `phase-X/flow.md`

通常走：

```
Phase 1 → Phase 2（可选）→ Phase 3 → Phase 4
```

---

## 大型需求：拆迭代

适用：完整子系统、跨多个领域、支付/权限/平台级能力。

先输出迭代规划，不直接进 Phase 1。

```markdown
## 迭代规划：<系统/功能名>

### 整体分析
<涉及领域、核心风险>

### 迭代拆分建议

| 迭代 | 范围 | 核心产出 | 预计切片数 | 依赖 |
|------|------|----------|------------|------|
| 迭代 1 | <范围> | <产出> | ~N | 无 |
| 迭代 2 | <范围> | <产出> | ~N | 迭代 1 |

### 建议
<为什么这样拆>
```

用户确认后，按迭代逐个走标准流程。

---

## 路由决策树

```
用户需求进入
  → 是开发/变更类需求吗？
      ├── 否 → routes/exception-routing.md#non-development
      └── 是
          → 是否已有功能变更 / 重构 / 原型反推？
              ├── 是 → routes/exception-routing.md
              └── 否 → 评估规模
                    ├── 微型 → 直接改
                    ├── 小型 → 计划模式
                    ├── 中型 → routes/phase-routing.md
                    └── 大型 → 迭代规划
```
