# Phase 3：Spec & 开发

> **定位：** Agent 自主开发的核心阶段。拆解切片、逐个 TDD 开发、测试自验，尽量不找人。
> **人机分工：** Agent 主导，人只在切片拆解确认和卡点排障时介入。
> **前置：** Phase 1 的 PRD + Phase 2 的架构设计/任务清单（如有）
> **产出：** 源代码 + 测试 + 自验通过

---

## 启动检查

### 0. Harness 预检（Java 项目）

如果项目根目录存在 `.harness/`，**优先加载**：

```
1. 读 .harness/ai-context/project-map.yaml    → 模块边界与依赖
2. 读 .harness/ai-context/business-rules.yaml → 幂等/缓存/一致性约束
3. 读 .harness/ai-context/coding-rules.yaml   → 编码规则
4. 如有 .harness/ai-context/coding-rules-*.yaml → 框架特化规则
5. 读 .harness/ai-context/error-catalog.yaml  → 错误码参考
```

如果文件为空模板 → 有 PRD 则提示提取，无 PRD 提示填写，拒绝则跳过并标注 `[上下文缺失]`。

### 1. 检查 Phase 2 是否存在

| 情况 | 处理 |
|------|------|
| Phase 2 已完成 → 有任务清单 | 读任务清单，按任务逐个执行 |
| Phase 2 被跳过 → 无任务清单 | Agent 从 PRD 直接拆切片，**第一个切片前执行「全局设计 fallback」** |

### 2. 全局设计 fallback（仅当 Phase 2 被跳过时）

```
全局设计（一次性，后续切片复用）
  ├── 技术栈确认（语言/框架/数据库/缓存/MQ）
  ├── 模块边界（如果是多模块项目）
  ├── 命名约定（包名/类名/API 路径前缀/响应格式）
  └── 共享 Entity（多切片共用的基础实体）
```

> **设计原则：** 全局设计只写「约束其他切片的设计决策」，不写单切片的实现细节。

### 3. 环境自检

每个切片开始前，检查本地环境：

```bash
.harness/devops/env-check.sh
```

环境不可用时：
- Docker 未运行 → 提示用户启动 Docker
- 容器未启动 → 执行 `docker compose -f .harness/devops/docker-compose.yml up -d`
- MySQL/Redis 连接失败 → 等待 10s 重试，仍失败则提示用户排查

## 流程总览

```
① 需求理解 → Agent 读 PRD + 上下文
    ↓
② 切片拆解 → Agent 拆切片 → 人快速确认
    ↓
③ 逐切片循环 ──────────────────────────┐
   ┌── Spec 分析（决策树）              │
   ├── 技术设计（API + Entity + DB）     │
   ├── TDD 五步（编译→单测→切片→集成→重构）│
   ├── 测试通过？── 是 → 推进下一切片 ──┘
   │                否 ↓
   ├── 自修循环（分析→修复→重跑）         │
   ├── 修好了？── 是 → 推进下一切片 ──┘
   │           否 ↓
   ├── 重试 ≥ 3 次？── 是 → 升级给人     │
   │                        ↓
   └── 人介入 → 排障 → 继续 ────────────┘
    ↓
④ 全部切片完成 → 自验
    ↓
⑤ 退出 → Phase 4 或下一任务
```

## ① 需求理解

Agent 读取 PRD + `.harness/ai-context/` 全部上下文，内部整理：

- 本迭代要交付什么
- 有哪些已知约束（幂等/缓存/一致性/错误码）
- 有哪些外部依赖（外部 API / MQ / 缓存）

不产出文档，Agent 内部消化。

---

## ② 切片拆解

### 拆解原则

| 原则 | 说明 |
|------|------|
| **每个切片可独立验证** | 切片完成后可以跑测试确认正确 |
| **切片之间有依赖顺序** | 先做底层（Entity/Repo），再做 Service，最后 Controller |
| **一个切片 = 一个可交付增量** | 不是按"写一个类"拆，是按"完成一个功能点"拆 |

### 拆解格式

Agent 拆完后，输出给用户确认：

```
【切片拆解】本迭代共 N 个切片

1. <切片名> — <一句话描述> — 依赖：无
2. <切片名> — <一句话描述> — 依赖：切片 1
3. <切片名> — <一句话描述> — 依赖：切片 1, 2
...

请确认切片划分是否合理，确认后我开始逐个开发。
```

### 人确认规则

- 切片划分合理 → 用户说「可以」「没问题」→ 开始逐切片开发
- 切片需要调整 → 用户提出修改 → Agent 调整后重新确认
- 简单迭代（1~3 个切片）→ Agent 拆完直接开始，不等确认

## ③ 逐切片开发

### Spec 决策树（阶段 ③Spec 分析）

核心原则从 `rules/decision-boundary.md`：

| 分支 | 触发条件 | 处理方式 |
|:----|------|------|
| **A：Agent 自定** | project.md 有约定 / 行业标准 / 项目已有同类参考 / 改动 ≤ 50 行 | 直接定，概要中一句告知 |
| **B：必须问用户** | 影响数据/范围/架构/外部系统 / PRD 完全没提 | 给 2~3 选项 + 建议 |
| **C：暂定 + 标记** | 既不满足 A 也不满足 B | 用最通用做法，标注 `[待确认]` |

决策点上限：分支 B 每轮 ≤ 3 个，超过的走分支 C。

### TDD 五步

如果项目存在 `.harness/templates/`，执行五步 TDD。
详见：`shared/tdd-five-steps.md`（入口/出口条件、命令、跳过规则、决策矩阵）。

```
① 编译 + 静态规则检查（≤ 3s）
② 单元测试（Mock 外部依赖）→ RED
③ 模块切片测试（只装配当前模块 Bean ≤ 30s）
④ 集成测试（Testcontainers 按需触发）
⑤ REFACTOR
```

**命令（Java Harness）：**

```bash
./gradlew fastTest          # 增量编译 + 切片测试，≤ 3s
./gradlew test              # 全量单元+切片测试，≤ 30s
./gradlew integrationTest   # 集成测试（Testcontainers），按需
```

生成代码时参考 `.harness/templates/` 下的模板。

## 测试失败处理：自修循环

测试不过时，**Agent 先自己修**，不第一时间找人。

### 自修流程

```
测试失败
    ↓
Agent 分析失败日志
    ├── 编译错误 → 修代码
    ├── 测试断言失败 → 检查实现是否符合 Spec，修代码
    ├── 环境问题 → 修环境（启动容器/改配置）
    └── 测试本身写错 → 修测试（只有确认是测试错才改测试）
    ↓
重新跑测试
    ├── 通过 → 推进下一切片 ✅
    └── 仍失败 → 回到分析（进入下一轮自修）
```

### 自修规则

1. **先分析再动手** — 读测试日志，理解失败原因，不要盲目试错
2. **改代码不改测试** — 除非确认测试本身有 bug
3. **每轮自修记录** — 内部记一下改了什么，避免重复尝试同一个方向
4. **不影响其他切片** — 修本切片时不能搞坏已有代码

### 升级触发条件

| 条件 | 动作 |
|------|------|
| 同一个测试连续修复 3 次仍失败 | 停下来，升级给人 |
| 测试失败原因超出代码层面（框架 bug / 依赖冲突 / 外部服务不可用） | 升级给人 |
| 修复方向不确定，无法定位根因 | 升级给人 |

升级时输出：

```
⚠️ 卡点：切片 <名称> 的 <测试名> 连续修复 N 次未通过。

失败原因分析：
<Agent 对失败原因的理解>

已尝试的修复：
1. <改了什么> → 结果
2. <改了什么> → 结果
3. <改了什么> → 结果

需要你帮忙判断方向。
```

## 切片间推进

一个切片 TDD + 自验通过后：

1. 更新 `shared/state.json`（标记切片完成）
2. 通报用户进度：「✅ 切片 N/M 完成：<名称>。继续下一切片：<名称>」
3. **不停下来等确认** — 直接开始下一切片（除非是第一个切片，让用户知道节奏）

例外：如果是第一个切片完成，简短通报后继续——让用户感受 Agent 在自主推进。

## 阶段 ④ 自验

> ⚠️ 这是自验，不是集成测试。Phase 4 才是集成测试。

| 维度 | Phase 3 自验 | Phase 4 集成测试 |
|------|------|------|
| **范围** | 本切片 + 已有代码 | 全量（所有切片 + 外部依赖） |
| **目的** | 验证本切片没搞坏别人 | 验证整个系统端到端可工作 |
| **产出** | 编译通过 + 已有测试不回归 | 集成测试报告 |

### 自验清单

- [ ] 本切片代码编译通过
- [ ] 本切片测试全部 GREEN
- [ ] 已有测试无回归（全量跑一遍）
- [ ] 如涉及 API：用 curl/脚本调通本接口
- [ ] 如涉及页面：页面可正常渲染

---

## 退出条件

全部切片 TDD 完成 + 自验通过 → 进入 Phase 4（集成测试）或下一个任务。

每完成切片 → 更新 `shared/state.json`。模板见 `shared/state.json`。
