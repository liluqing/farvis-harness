# Spec 决策树 — 判断标准

> Agent 在 Spec 阶段对每个决策点进行分类的依据。与 Phase 1 决策树同构，但判断标准是技术层面。

## 分支 A：Agent 自主决策

**满足任一条件即可走分支 A。Agent 直接定，一句话在概要中告知用户。**

| # | 条件 | 判断方法 | 示例 |
|:-:|------|------|------|
| A1 | **项目上下文中有明确约定** | 读 `shared/project-context.md` 或 `.harness/ai-context/coding-rules.yaml` | 表名单数/复数、API 前缀、命名风格 |
| A2 | **行业有广泛接受的标准** | 是否属于 RESTful/RFC/框架约定 | HTTP 状态码、Bean Validation、REST 资源命名 |
| A3 | **项目已有类似实现可参考** | 搜索已有代码是否有同模式 | 已有 Controller 的异常处理方式、分页格式 |
| A4 | **做错后改动成本低** | 改动 ≤ 50 行代码，不涉及数据迁移 | 变量命名、方法拆分、日志格式 |

## 分支 B：必须问用户

**满足任一条件即走分支 B。给出 2~3 个选项 + 建议。**

| # | 条件 | 判断方法 | 示例 |
|:-:|------|------|------|
| B1 | **影响用户数据或业务流程** | 决策涉及数据的创建/修改/删除策略 | 软删除 vs 硬删除、级联策略、数据保留 |
| B2 | **影响交付范围** | 这个版本做不做某个子功能 | 头像上传、导出 Excel、消息通知 |
| B3 | **引入新依赖或架构变化** | 需要加新的库/中间件/服务 | 选 Redis 还是本地缓存、消息队列还是同步 |
| B4 | **和外部系统交互** | 决策涉及第三方 API、外部服务 | 支付接口选型、短信服务商 |
| B5 | **PRD 完全没提且无法推断** | 搜索 PRD + 项目上下文均无相关信息 | 特殊业务规则、合规要求 |

## 分支 C：暂定 + 标记（新增）

**既不满足 A 也不满足 B 的灰色决策。用最通用/最保守的做法暂定，标注 `[待确认]`。**

| 场景 | 暂定做法 | 标注示例 |
|------|---------|------|
| 字段长度不确定 | 取宽松值（如 VARCHAR(255)） | `[待确认] 字段 X 暂用 VARCHAR(255)，确认业务约束后可收紧` |
| 索引策略不确定 | 只加主键和 FK 索引 | `[待确认] 暂不加组合索引，根据查询模式后续补充` |
| 缓存策略不确定 | 不用缓存 | `[待确认] 暂不引入缓存，性能压测后按需加` |
| 分页策略不确定 | 偏移分页（最通用） | `[待确认] 暂用 OFFSET/LIMIT 分页，大数据量可改游标` |
| 日志级别不确定 | INFO | `[待确认] 关键操作暂用 INFO 级别日志` |

**规则：**
- 分支 C 不限数量，但每个都要有「暂定选了什么 + 为什么」
- 在 Spec 概要中统一列出「暂定待确认」区
- 用户确认概要时一次性处理

## 决策点上限

- 分支 B 问题每轮 ≤ **3 个**
- 超过 3 个 → 优先问数据策略/架构选型，其余走分支 C

## Agent 自检清单

对每个决策点，自问：

```
1. 项目上下文中有约定吗？（project-context.md / coding-rules.yaml）
                                  → 有 → A, 跳过后续
2. 是行业标准做法吗？                      → 是 → A, 跳过后续
3. 做错改动成本 > 50 行代码或涉及数据迁移？  → 否 → A
                                          → 是 → 继续判断
4. 影响数据/范围/架构/外部系统吗？          → 是 → B
5. PRD 完全没提吗？                        → 是 → B
6. 以上都不满足？                          → 是 → C（暂定 + 标记）
```
