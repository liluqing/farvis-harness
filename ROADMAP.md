## 当前状态

**Step 1 已完成（v0.5.0）**：Skill 改造 + 异常场景处理机制 + 文档管理体系 + 整合。

已完成三轮改造：
- **第一轮**：Skill 从"Agent 自循环引擎"改为"人机协作操作手册"（1.1~1.4）
- **第二轮**：补充异常场景处理——会话初始化协议、需求规模四级评估、异常路由、文档健康度检查、跨迭代影响分析、前置条件不满足处理
- **第三轮**：文档管理体系——四层信息模型（AI-CONTEXT.md / project/ / ai-context/*.yaml / iterations+archive）、迭代目录结构、归档流程、Git Hook 事件信箱、Agent 交互协议、新旧体系整合

仓库同时已完成**静态基础设施层**（v0.3.1）：上下文 schema、代码模板、测试基础设施、Git hooks、Docker 环境——81 项资产全部就位。

**但这些资产还没被真正使用过。** 没有拿一个真实迭代从头到尾跑过一遍，所以不知道：
- 哪些模板 Agent 真正会读、能照着写
- 哪些编码规则 Agent 实际遵守了
- 人在哪个环节花时间最多
- Agent 在哪个环节产出质量不行
- 异常场景处理机制（会话初始化、四级路由、文档健康度）在实战中是否生效
- 文档管理体系（迭代目录、归档流程、AI Context 同步）在实战中是否顺畅

## 三步走

```
Step 1: 流程对齐（Skill 改造）        ← 把"操作手册"改对
    ↓
Step 2: 实战验证（真迭代跑一遍）        ← 暴露真问题
    ↓
Step 3: 针对性优化（按卡点改）         ← 有数据再动手
```

不做预先的全面重构，**先跑一遍再改。**

## Step 1: 流程对齐 — Skill 改造 + 文档管理体系 ✅

**目标**：让 harness-dev-flow Skill 从"Agent 自循环引擎"改为"人机协作操作手册"，并建立完整的文档管理体系。

| # | 改造项 | 具体内容 | 完成标准 | 状态 |
|:--|------|---------|---------|:--:|
| 1.1 | Phase 定义改为迭代模型 | 从线性 Phase 1→4 改为**迭代式**：一个迭代 = 需求→拆解→开发→验证→交付的完整闭环 | Skill 中明确描述迭代生命周期，人看完知道"一个迭代该怎么跑" | ✅ |
| 1.2 | 每个 Phase 标注人机分工 | 每个环节标注：谁做（人/Agent/人机协同）、人需要看什么、Agent 自主到什么程度 | 人看完知道"这个环节我要不要介入" | ✅ |
| 1.3 | Agent 自主行为边界写清 | Agent 在什么条件下自己往下跑、什么条件下必须停下问人 | Agent 读完知道"什么时候我可以继续、什么时候该停" | ✅ |
| 1.4 | 去掉自动编排的假设 | 删除 Skill 中隐含的"Agent 会自动推进到下一阶段"的措辞 | 每个阶段之间都是人触发，不假装自动 | ✅ |
| 1.5 | 会话初始化协议 | 每个新会话自动恢复项目上下文（读 state.json + ai-context + 文档） | Agent 醒来不会"失忆"，能自动衔接上次进度 | ✅ |
| 1.6 | 需求规模四级评估 | 微型(直接改) / 小型(计划模式) / 中型(标准流程) / 大型(拆迭代) | 用户一句话想搞整个系统时，Agent 会先评估规模再路由 | ✅ |
| 1.7 | 异常路由表 | 处理用户跳步：从原型反推 PRD、变更模式、技术重构模式、非开发问题识别 | 用户不按标准流程走时 Agent 有应对方案 | ✅ |
| 1.8 | 文档健康度检查 | 每个 Phase 启动前检查 PRD/架构/状态文件一致性 | 防止基于过时文档做决策 | ✅ |
| 1.9 | 跨迭代影响分析 | 修改已有模块时评估影响范围 | 迭代 N+1 改迭代 N 的模块时不会遗漏依赖方 | ✅ |
| 1.10 | 文档管理体系 | 四层信息模型 + 迭代目录结构 + 归档流程 + Git Hook 事件信箱 | 迭代文档集中管理，归档后自动同步 AI Context | ✅ |
| 1.11 | 新旧体系整合 | flow/skill.md + SKILL.md 更新，统一路径和 Skill 加载关系 | Agent 执行时不再有两套冲突的文档路径 | ✅ |

**不做的事**：不写编排器脚本、不做自动化状态机、不做 Agent 自循环逻辑。

## Step 2: 实战验证 — 真迭代跑一遍 ✅

**目标**：拿一个真实功能模块，走完一个完整迭代，记录所有卡点。

**验证迭代**：`2026-06-19_积分体系_v1.0`（已归档）

| # | 验证项 | 具体内容 | 完成标准 | 状态 |
|:--|------|---------|---------|:--:|
| 2.1 | 选定验证目标 | 从 Farvis-AI PRD 中选积分体系模块，定义为一个迭代 | 有明确的迭代目标和验收标准 | ✅ |
| 2.2 | 走完完整迭代 | 需求确认 → 技术设计 → 后端开发 → 前端开发 → 联调验证 → 归档 | 每一步都记录：Agent 做了什么、人介入了什么、卡在哪里 | ✅ |
| 2.3 | 记录卡点清单 | 用结构化格式记录每个卡点 | 每个卡点有：环节、现象、原因分析、改进方向 | ✅ |
| 2.4 | 评估资产有效性 | 记录哪些 .harness/ 资产被 Agent 真正使用了 | 有清单：用了什么、没用什么、缺什么 | ✅ |

**卡点摘要（详见 [harness-v0.6-optimization-plan.md](Docs/harness-v0.6-optimization-plan.md)）**：

| 类别 | 卡点数 | 核心问题 |
|------|:------:|---------|
| 环境配置 | 3 | 密码散落在多处、环境变量脱敏、Maven 参数语义差异 |
| 需求/设计 | 3 | 前端缺登录流程、无 Register 接口、文档与代码不一致 |
| 流程协作 | 3 | 阶段转换等指令、trivial fixes 还在确认、多仓库协调 |
| 归档 | 2 | 流程偏重、文档合并手工操作多 |

**关键发现**：用户介入 10 次中，6 次是不必要的（Agent 可自决）。根因是缺少明确的"Agent 决策标准矩阵"。

**产出**：harness-v0.6-optimization-plan.md（7 项改造方案，P0~P3 优先级排序）

## 待优化项（实战验证发现）

> 以下改进点来自 Step 2 实战验证，在 Step 3 针对性优化阶段处理。
> 
> **详细方案**：[harness-v0.6-optimization-plan.md](Docs/harness-v0.6-optimization-plan.md)

| # | 改进点 | 来源 | 优先级 | 说明 |
|---|--------|------|--------|------|
| 1 | **PRD 阶段必须输出原型** | 积分体系迭代 | P0 | 用户提供则直接用，未提供则由 Agent 生成 HTML 原型。原型是 PRD 的视觉规格基准，后续页面清单和技术设计都基于原型对齐。 |
| 2 | **多角色 Review 机制** | 积分体系迭代 | P1 | 已沉淀为 `prd-role-review` Skill，但需集成到标准流程（Phase 1 完成后自动触发）。**关键改进：给 subagent 传完整文档路径（PRD + 原型 + 技术设计），不要自己摘摘要——信息损失会导致 review 质量下降。** |
| 3 | **PRD 结构优化** | 积分体系迭代 | P1 | 页面清单应放在 PRD 开头（第 1 章），而非末尾。让读者先建立视觉认知，再看功能细节。 |
| 4 | **Agent 决策标准矩阵** | 积分体系迭代 | P0 | 用户介入 10 次中 6 次不必要。定义清晰的决策矩阵：能跑测试验证的事 Agent 自己干，不能跑测试验证的事问人。 |
| 5 | **环境配置标准化** | 积分体系迭代 | P0 | 密码散落在 docker-compose.yml 和 application.yml，3 轮试错。新增 dev-env.yaml 统一管理。 |
| 6 | **横切关注点清单** | 积分体系迭代 | P1 | 前端缺登录流程、无 Register 接口。Phase 2 技术设计时强制检查认证、日志、异常处理、测试数据初始化。 |
| 7 | **文档同步前置** | 积分体系迭代 | P1 | 归档时发现 tech-design.md 和代码不一致。改为每个切片完成时强制同步 api-changes.md 和 ddl-changes.md。 |
