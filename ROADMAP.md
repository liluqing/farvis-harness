## 三步走

```
Step 1: 流程对齐（Skill 改造）        ← 把"操作手册"改对
    ↓
Step 2: 实战验证（真迭代跑一遍）        ← 暴露真问题
    ↓
Step 3: 针对性优化（按卡点改）         ← 有数据再动手
```

不做预先的全面重构，**先跑一遍再改。**

## Step 1: 流程对齐 — Skill 改造 + 文档管理体系 ✅

**目标**：让 harness-dev-flow Skill 从"Agent 自循环引擎"改为"人机协作操作手册"，并建立完整的文档管理体系。

| # | 改造项 | 具体内容 | 完成标准 | 状态 |
|:--|------|---------|---------|:--:|
| 1.1 | Phase 定义改为迭代模型 | 从线性 Phase 1→4 改为**迭代式**：一个迭代 = 需求→拆解→开发→验证→交付的完整闭环 | Skill 中明确描述迭代生命周期，人看完知道"一个迭代该怎么跑" | ✅ |
| 1.2 | 每个 Phase 标注人机分工 | 每个环节标注：谁做（人/Agent/人机协同）、人需要看什么、Agent 自主到什么程度 | 人看完知道"这个环节我要不要介入" | ✅ |
| 1.3 | Agent 自主行为边界写清 | Agent 在什么条件下自己往下跑、什么条件下必须停下问人 | Agent 读完知道"什么时候我可以继续、什么时候该停" | ✅ |
| 1.4 | 去掉自动编排的假设 | 删除 Skill 中隐含的"Agent 会自动推进到下一阶段"的措辞 | 每个阶段之间都是人触发，不假装自动 | ✅ |
| 1.5 | 会话初始化协议 | 每个新会话自动恢复项目上下文（读 state.json + ai-context + 文档） | Agent 醒来不会"失忆"，能自动衔接上次进度 | ✅ |
| 1.6 | 需求规模四级评估 | 微型(直接改) / 小型(计划模式) / 中型(标准流程) / 大型(拆迭代) | 用户一句话想搞整个系统时，Agent 会先评估规模再路由 | ✅ |
| 1.7 | 异常路由表 | 处理用户跳步：从原型反推 PRD、变更模式、技术重构模式、非开发问题识别 | 用户不按标准流程走时 Agent 有应对方案 | ✅ |
| 1.8 | 文档健康度检查 | 每个 Phase 启动前检查 PRD/架构/状态文件一致性 | 防止基于过时文档做决策 | ✅ |
| 1.9 | 跨迭代影响分析 | 修改已有模块时评估影响范围 | 迭代 N+1 改迭代 N 的模块时不会遗漏依赖方 | ✅ |
| 1.10 | 文档管理体系 | 四层信息模型 + 迭代目录结构 + 归档流程 + Git Hook 事件信箱 | 迭代文档集中管理，归档后自动同步 AI Context | ✅ |
| 1.11 | 新旧体系整合 | flow/skill.md + SKILL.md 更新，统一路径和 Skill 加载关系 | Agent 执行时不再有两套冲突的文档路径 | ✅ |

**不做的事**：不写编排器脚本、不做自动化状态机、不做 Agent 自循环逻辑。

## Step 2: 实战验证 — 真迭代跑一遍

**目标**：拿一个真实功能模块，走完一个完整迭代，记录所有卡点。

| # | 验证项 | 具体内容 | 完成标准 |
|:--|------|---------|---------|
| 2.1 | 选定验证目标 | 从 Farvis-AI PRD 中选一个功能模块（如视频生成接口），定义为一个迭代 | 有明确的迭代目标和验收标准 |
| 2.2 | 走完完整迭代 | 需求确认 → Agent 拆解切片 → 逐切片开发 → 测试验证 → 集成验证 → 交付 | 每一步都记录：Agent 做了什么、人介入了什么、卡在哪里 |
| 2.3 | 记录卡点清单 | 用结构化格式记录每个卡点 | 每个卡点有：环节、现象、原因分析、改进方向 |
| 2.4 | 评估资产有效性 | 记录哪些 .harness/ 资产被 Agent 真正使用了 | 有清单：用了什么、没用什么、缺什么 |

**这一步的产出决定 Step 3 做什么。** 不预设改进方向。

## Step 3: 针对性优化 — 按卡点改

**目标**：根据 Step 2 暴露的真问题，针对性地改。这一步的具体内容在 Step 2 完成后再定义。

预判可能的改进方向（仅为参考，实际以卡点清单为准）：

- **Agent 读不懂上下文** → 改 ai-context schema 的格式/粒度
- **Agent 不遵守编码规则** → 把规则从 YAML 提取到 CLAUDE.md/AGENTS.md 的更显眼位置
- **Agent 测试跑不过后不会自己修** → Skill 中补"测试失败→分析→修复"的行为指导
- **人在某个环节卡太久** → 该环节的 Skill 指导写得不够清晰，补充操作步骤
- **缺少某个模板** → 补模板
