## 2. 需求背景

Farvis-AI 采用 Credits（积分）作为统一消费单位。所有核心操作（视频生成、数字人创建、语音克隆）都消耗 Credits。本迭代实现积分体系的完整闭环：

- 用户能**看到自己有多少 Credits**
- 用户能**花钱买 Credits**（支付）
- 系统能**正确扣减 Credits**（消费）
- 用户能**看到每一笔消费记录**

这是整个平台的**商业基础设施**——没有积分体系，其他功能的商业化无法闭环。

## 3. 用户场景

| 场景 | 用户角色 | 操作流程 | 预期结果 |
|------|---------|---------|---------|
| A. 查看余额 | 已登录用户 | 侧栏 → 积分中心 | 醒目展示当前 Credits 余额 + 当前套餐 |
| B. 购买套餐 | 免费/付费用户 | Header "购买会员" / 积分中心选套餐 → 支付 | Credits 即时到账，套餐状态更新 |
| C. Credits 不足 | 已登录用户 | 创作页提交 → 余额不足 → 跳转积分中心 | 明确提示 + 引导充值 |
| D. 查看消费记录 | 已登录用户 | 积分中心 → 使用记录 | 按时间倒序列出每笔消费/充值明细 |
| E. 免费体验 | 新注册用户 | 注册完成 | 自动赠送 50 Credits（一次性） |

## 5. 业务规则

| 规则 | 描述 |
|------|------|
| R1 消费标准 | 视频生成 = 50C，Avatar 创建 = 100C，Voice 克隆 = 50C |
| R2 套餐定价 | 入门版 ¥99/月(500C)，专业版 ¥299/月(2000C)，企业版 ¥999/月(10000C) |
| R3 扣减原子性 | Credits 扣减必须与业务操作在同一事务中完成（**演示版简化：单线程，无需分布式锁**） |
| R4 余额不足 | 余额 < 所需 Credits 时，拒绝操作，返回 402，不调用下游（HeyGen） |
| R5 幂等 | 同一模拟支付订单重复点击时，仅入账一次 |
| R6 新用户赠送 | 注册即送 50 Credits，一次性，不重复发放 |
| R7 消费记录 | 每笔 Credits 变动必须写入流水表，不可只改余额 |
| R8 Credits 有效期 | **购买后永久有效**（不月底清零） |
| R9 套餐有效期 | **套餐无有效期**（演示版简化，不涉及到期/续费） |

## 6. 验收标准

### 核心功能验收
- [ ] 新用户注册后余额 = 50 Credits
- [ ] 积分中心页面正确展示余额、当前套餐、三个套餐卡片
- [ ] 点击套餐「选择」→ 弹出模拟支付弹窗 → 点击「确认支付」→ Credits 即时到账（延迟 ≤1s）
- [ ] 点击「取消」→ 关闭弹窗，Credits 不变
- [ ] 余额充足时 → 正确扣减（视频-50C / Avatar-100C / Voice-50C）
- [ ] 余额不足时提交视频生成 → 返回 402 + "Credits 不足，去充值" + 跳转积分中心
- [ ] 消费记录按时间倒序展示，每条含操作类型、Credits 变动（消费红/充值绿）、时间
- [ ] 同一模拟支付订单重复点击 → 仅入账一次（幂等）

### 数据一致性验收
- [ ] 所有 Credits 变动（消费/充值/赠送）均写入流水表
- [ ] 余额 = 累计充值 - 累计消费（可从流水表计算验证）

### 不验收项（演示版简化）
- ❌ 真实支付宝/微信支付流程
- ❌ 支付超时自动取消
- ❌ 支付失败补单机制
- ❌ 套餐升降级
- ❌ 套餐到期处理

## 7. 非功能需求

| 维度 | 要求 |
|------|------|
| 性能 | 余额查询 < 100ms，消费扣减 < 200ms |
| 一致性 | Credits 扣减与业务操作强一致（同一事务） |
| 幂等 | 模拟支付接口幂等（重复点击不入账） |
| 审计 | 所有 Credits 变动有完整流水，可追溯 |

> **演示版简化说明**：
> - 无需考虑支付安全（签名校验、防重放）
> - 无需考虑高并发（分布式锁、乐观锁）
> - 无需考虑支付渠道对账
