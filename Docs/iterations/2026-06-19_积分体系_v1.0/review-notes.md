# 积分体系 PRD Review — Staff PM 评审意见

> 评审人：模拟 Google Staff PM（Cloud Billing / Play Subscriptions / Google One Credits）
> 评审日期：2026-06-19
> 总体评价：**PRD 框架清晰，但在"订阅生命周期"和"合规前置"两个维度存在致命遗漏，不建议直接进入技术设计阶段。**

---

## 一、Critical（阻塞级，必须在编码前解决）

| # | 维度 | 问题 | 说明 | 影响 |
|---|------|------|------|------|
| C1 | 产品闭环 | **订阅 vs 一次性购买：核心商业模型未定义** | 套餐标注"¥99/**月**"，但支付流程描述的是一次性购买（选套餐→支付→到账）。到底是**自动续费订阅**还是**每月手动购买**？这决定了整个系统的骨架：是否需要订阅管理（取消/暂停/换绑支付方式）、是否需要续费提醒、是否需要处理续费失败。**这不是细节，是产品灵魂。** | 若上线后发现理解错误，全部支付+账务逻辑要推倒重写 |
| C2 | 产品闭环 | **Credits 过期/清零策略完全空白** | PRD 说"跨月累积策略"是 Out of Scope，但这恰恰是用户**必然追问**的问题：月初 500C 用了 200C，月底剩 300C 是清零还是累积？如果"不定义"就上线，用户充完值发现月底清零，会引发**大量投诉和退款请求**。即使不做，也必须在 PRD 中写明"本期 Credits 月底清零/不累积"并让用户在购买前看到该条款。 | 直接涉及收入确认合规和用户信任 |
| C3 | 合规风险 | **支付资质和商户号未提及** | 接入支付宝和微信支付需要：① 企业营业执照 ② ICP 备案 ③ 支付宝/微信商户号申请与签约 ④ 支付接口权限开通（微信支付 Native 支付、支付宝当面付/手机网站支付）。**这些流程通常需要 1-4 周**，且需要法人签字和对公打款验证。PRD 零提及。 | 没有商户号，支付功能无法上线。这是最硬的卡点 |
| C4 | 合规风险 | **发票（税务合规）完全缺失** | 中国市场：用户付费后有权索取发票（增值税普通发票或专用发票）。¥99-¥999 的价位段，个人用户和企业用户都会问。PRD 没有任何发票相关流程。**这不只是体验问题，是税务合规问题。** | 企业用户无法报销 → 直接流失；税务稽查风险 |
| C5 | 合规风险 | **自动续费合规（若为订阅制）** | 若 C1 的答案是"自动续费订阅"，则必须遵守：①《网络交易监督管理办法》（2021）要求自动续费前 5 日显著提醒 ② 2024 年工信部 APP 整治要求"一键取消订阅" ③ 微信支付/支付宝的自动扣款签约需要单独的签约流程。PRD 零覆盖。 | 违规可能被下架整顿、罚款 |
| C6 | 产品闭环 | **套餐升降级/变更流程缺失** | 用户在月中想从入门版(¥99)升级到专业版(¥299)，怎么办？是补差价？还是新购叠加？原套餐剩余 Credits 怎么处理？降级呢？**这是付费用户最常见的操作之一。** | 没有升降级 → 用户只能等月底 → 体验差 → 降级为只买最低档 |
| C7 | 产品闭环 | **套餐到期后的状态机未定义** | 套餐到期后：① 用户状态变成什么？"免费用户"？② 剩余 Credits 怎么办？③ 能否继续消费？④ 如果 HeyGen 上游有正在执行的任务，套餐到期了怎么处理？ | 到期处理不当 = 用户正在生成视频突然中断 = 灾难级体验 |

## 二、Major（重要，建议在编码前或编码中解决）

| # | 维度 | 问题 | 说明 |
|---|------|------|------|
| M1 | 遗漏场景 | **并发扣减竞态条件**：用户同时在两个浏览器 tab 各提交一个视频生成请求，余额只剩 50C。两次请求几乎同时到达服务端，都通过了余额检查，各扣 50C → 余额变成 -50C。PRD 只说"同一事务"，但没说如何防并发。需要分布式锁或乐观锁。 |
| M2 | 遗漏场景 | **HeyGen 调用失败但 Credits 已扣**：R3 要求"扣减与业务操作同一事务"，但 HeyGen 是外部 API，无法纳入本地事务。如果 Credits 先扣了，HeyGen 返回 500 或超时，Credits 要不要退？PRD 没有定义这个回滚策略。这是**分布式事务的经典难题**。 |
| M3 | 遗漏场景 | **消费记录分页和筛选**：重度用户可能有数百条记录。PRD 只说"时间倒序列表"，没有分页、没有筛选（按类型/时间范围）、没有搜索。消费记录是用户对账和信任的基础。 |
| M4 | 用户旅程 | **支付失败/取消后的引导缺失**：用户在支付弹窗选了支付宝，跳转后取消支付或支付失败，回到 App 后看到什么？是回到积分中心？还是弹出"支付未完成"的提示？PRD 只定义了"超时自动取消"，没有定义用户主动取消或支付失败的回流体验。 |
| M5 | 用户旅程 | **Header "购买会员" 与积分中心的一致性**：场景 B 提到 Header 有"购买会员"按钮，但积分中心页面叫"积分中心"。这两个入口的落地页是同一个吗？如果是，为什么叫"购买会员"而不是"购买积分"？命名混乱会让用户困惑："会员"和"Credits"是什么关系？ |
| M6 | 商业风险 | **定价未定就进开发**：补充上下文明确说"定价是否为最终定价（未定）"。如果定价在开发过程中变更，前端硬编码的三档套餐、后端扣减逻辑、支付金额校验全要改。**建议套餐配置做成后端可配，不要硬编码。** |
| M7 | 商业风险 | **企业版 ¥999/月的购买流程不适合自助**：¥999/月 = ¥11,988/年。这个价位的企业客户通常需要：① 对公转账 ② 合同签署 ③ 增值税专用发票 ④ 客户成功对接。让他们像个人用户一样扫码支付是不现实的。建议企业版改为"联系我们"而非"选择"。 |
| M8 | 技术风险 | **补单机制"≤3次"缺乏细节**：R7 说"自动补单≤3次"，但：① 补单间隔是多少？（指数退避？固定间隔？）② 3次之间的时间窗口是多久？③ "创建客服工单"——用什么系统？Jira？飞书？手动？④ 客服处理后怎么手动补 Credits？有管理后台吗？ |
| M9 | 合规风险 | **用户协议 / 购买须知缺失**：用户付款前必须看到并同意：① Credits 使用规则（有效期、不可转让等）② 退款政策 ③ 自动续费条款（若适用）。PRD 没有提及任何协议页面或勾选框。 |
| M10 | 遗漏场景 | **未成年人支付保护**：面向中国市场的 C 端产品，如果用户中有未成年人，需要：① 实名认证 ② 支付限额 ③ 监护人同意。虽然 B2B 工具可能不涉及，但 PRD 应明确声明目标用户年龄段。 |

## 三、Minor（建议优化，可后续迭代）

| # | 维度 | 问题 | 说明 |
|---|------|------|------|
| m1 | 用户体验 | **余额展示缺乏实时性说明**：用户同时在创作页和积分中心，创作页消耗了 Credits，积分中心的余额是否自动刷新？需要定义刷新策略（轮询/WebSocket/手动刷新）。 |
| m2 | 用户体验 | **消费记录缺少关联跳转**：消费记录里"视频生成 -50C"，用户可能想知道是哪个视频。建议记录中关联具体的资源 ID，支持跳转到对应资源。 |
| m3 | 用户体验 | **新用户 50C 用完后的引导策略**：新用户 50C 只够生成 1 个视频。用完后如果没有任何引导（push/邮件/站内信），用户可能直接流失。建议定义"余额低于 X 时提醒"的策略。 |
| m4 | 产品闭环 | **"本月使用"的时间窗口定义**："本月"是自然月（1号-月底）还是按用户注册日/购买日计算的 30 天周期？需明确。 |
| m5 | 技术优化 | **Apple/Google Play 渠道预留**：如果未来有移动端 APP，Apple 抽成 30%。当前定价体系可能需要调整。建议在架构上预留多渠道支付扩展能力。 |
| m6 | 运营风险 | **无对账机制**：PRD 没有定义支付渠道与内部 Credits 系统的日终对账流程。长期运营中，差异是必然出现的（网络超时、渠道侧异常等），需要 T+1 自动对账 + 差异告警。 |
| m7 | 安全 | **风控策略空白**：没有提及：① 同一用户短时间大量购买（可能是盗刷信用卡）② 异常 IP/设备的购买行为 ③ Credits 转赠/倒卖防范。建议至少定义基础风控规则。 |

## 五、总结与行动建议

### ✅ PRD 做得好的部分
- 页面清单和组件拆分清晰
- 业务规则表（R1-R9）覆盖了核心逻辑
- 异常处理有基本思考（幂等、补单、超时）
- 验收标准可量化

### ❌ 核心问题
1. **商业模型模糊**：订阅 vs 一次性购买是整个系统的根基，PRD 回避了这个决策
2. **合规前置事项全部缺失**：支付资质、发票、用户协议、自动续费合规——这些不是"以后再说"的事
3. **套餐生命周期不完整**：只有"购买"，没有"续费/升级/降级/到期/取消"
4. **分布式事务处理过于简化**：HeyGen 外部 API + Credits 扣减的失败回滚没有可行方案

### 🎯 建议的下一步
1. **暂停技术设计**，先用 1-2 天补齐上述 C1-C7 的产品决策
2. **同步启动**支付商户号申请和法务协议起草（这两项耗时最长）
3. 决策完成后，**更新 PRD 至 v1.1**，补充订阅生命周期状态机和套餐变更流程
4. 然后才进入技术设计阶段
