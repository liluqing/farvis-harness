# 产品需求文档（PRD）

> **迭代**：2026-06-19_积分体系_v1.0
> **作者**：AI（基于 Farvis-AI PRD v1.3）
> **状态**：草稿 → 待 review

---

## 1. 需求背景

Farvis-AI 采用 Credits（积分）作为统一消费单位。所有核心操作（视频生成、数字人创建、语音克隆）都消耗 Credits。本迭代实现积分体系的完整闭环：

- 用户能**看到自己有多少 Credits**
- 用户能**花钱买 Credits**（支付）
- 系统能**正确扣减 Credits**（消费）
- 用户能**看到每一笔消费记录**

这是整个平台的**商业基础设施**——没有积分体系，其他功能的商业化无法闭环。

---

## 2. 用户场景

| 场景 | 用户角色 | 操作流程 | 预期结果 |
|------|---------|---------|---------|
| A. 查看余额 | 已登录用户 | 侧栏 → 积分中心 | 醒目展示当前 Credits 余额 + 当前套餐 |
| B. 购买套餐 | 免费/付费用户 | Header "购买会员" / 积分中心选套餐 → 支付 | Credits 即时到账，套餐状态更新 |
| C. Credits 不足 | 已登录用户 | 创作页提交 → 余额不足 → 跳转积分中心 | 明确提示 + 引导充值 |
| D. 查看消费记录 | 已登录用户 | 积分中心 → 使用记录 | 按时间倒序列出每笔消费/充值明细 |
| E. 免费体验 | 新注册用户 | 注册完成 | 自动赠送 50 Credits（一次性） |

## 3. 功能范围

### 3.1 本期实现（In Scope）

| 功能点 | 优先级 | 说明 |
|--------|:------:|------|
| Credits 余额查询 | P0 | 醒目展示当前余额 + 当前套餐名 |
| 套餐展示与选择 | P0 | 三档套餐卡片，当前套餐高亮 |
| 支付下单（支付宝） | P0 | 选择套餐 → 创建订单 → 跳转支付宝 → 回调到账 |
| 支付下单（微信） | P0 | 同上，微信支付渠道 |
| Credits 消费扣减 | P0 | 被其他模块调用：视频-50C、Avatar-100C、Voice-50C |
| 消费记录查询 | P1 | 按时间倒序，每条含类型/描述/变动/时间 |
| 新用户注册赠送 | P0 | 注册时一次性赠送 50 Credits |
| 支付异常处理 | P0 | 超时取消、补单重试、幂等检测 |

### 3.2 本期不实现（Out of Scope）

| 功能点 | 原因 |
|--------|------|
| 退款 | v1.0 不支持，需人工客服 |
| Credits 跨月累积策略 | 待商业决策确认（PRD §15 待确认项） |
| 促销/活动赠送 Credits | "领会员""领福利"入口为运营功能，后续迭代 |
| API 对外计费 | API 为 v1.1+ 规划 |
| 企业批量采购 | 企业服务为人工跟进模式 |

## 4. 业务规则

| 规则 | 描述 |
|------|------|
| R1 消费标准 | 视频生成 = 50C，Avatar 创建 = 100C，Voice 克隆 = 50C |
| R2 套餐定价 | 基础版 ¥99/月(500C)，专业版 ¥399/月(2000C)，企业版 ¥999/月(10000C) |
| R3 扣减原子性 | Credits 扣减必须与业务操作在同一事务中完成，不允许"扣了钱没生成"或"生成了没扣钱" |
| R4 余额不足 | 余额 < 所需 Credits 时，拒绝操作，返回 402，不调用下游（HeyGen） |
| R5 订单超时 | 支付订单 15 分钟未支付自动取消 |
| R6 幂等 | 同一订单重复支付时，仅入账一次 |
| R7 补单 | 支付成功但未到账时，自动补单 ≤3 次，仍失败则创建客服工单 |
| R8 新用户赠送 | 注册即送 50 Credits，一次性，不重复发放 |
| R9 消费记录 | 每笔 Credits 变动（消费/充值/赠送/退款）必须写入流水表，不可只改余额 |

---

## 5. 验收标准

- [ ] 新用户注册后余额 = 50 Credits
- [ ] 积分中心页面正确展示余额、当前套餐、三个套餐卡片（当前套餐高亮）
- [ ] 选择套餐 → 支付宝支付成功 → Credits 即时到账（延迟 ≤3s）
- [ ] 选择套餐 → 微信支付成功 → Credits 即时到账
- [ ] 余额不足时提交视频生成 → 返回 402 + "Credits 不足，去充值" + 跳转积分中心
- [ ] 余额充足时 → 正确扣减（视频-50C / Avatar-100C / Voice-50C）
- [ ] 消费记录按时间倒序展示，每条含操作类型、描述、Credits 变动（消费红/充值绿）、时间
- [ ] 支付订单 15 分钟未支付 → 自动取消
- [ ] 同一订单重复支付 → 仅入账一次（幂等）
- [ ] 支付成功未到账 → 自动补单 ≤3 次

---

## 6. 非功能需求

| 维度 | 要求 |
|------|------|
| 性能 | 余额查询 < 100ms，消费扣减 < 200ms |
| 一致性 | Credits 扣减与业务操作强一致（同一事务或可靠最终一致） |
| 安全 | 支付签名校验、金额服务端校验、防重放 |
| 幂等 | 所有涉及 Credits 变动的接口必须幂等 |
| 审计 | 所有 Credits 变动有完整流水，可追溯 |
