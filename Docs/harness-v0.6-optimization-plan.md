# Harness v0.6.0 改造方案

> 基于 2026-06-19 积分体系迭代的实战验证，暴露的卡点和优化方向。
>
> **一句话原则：能跑测试验证的事 Agent 自己干，不能跑测试验证的事问人。**

---

## 1. 卡点清单

### 1.1 环境配置类

| # | 卡点 | 现象 | 根因 | 影响 |
|:-:|------|------|------|------|
| E1 | 数据库密码不一致 | `application.yml` 密码和 Docker 容器不匹配，3 轮试错 | 没有统一的环境配置文档，密码散落在 docker-compose.yml 和 application.yml | 浪费 ~10 分钟 |
| E2 | 环境变量脱敏 | `docker inspect` 和 `docker exec env` 被 Hermes 安全扫描脱敏成 `***` | Hermes 安全策略，无法关闭 | 排查困难，只能暴力枚举 |
| E3 | 测试编译阻塞启动 | `-DskipTests` 仍编译测试代码，import 缺失时 build 失败 | Maven 两个参数语义差异（`-DskipTests` vs `-Dmaven.test.skip=true`） | Agent 不知道正确参数 |

### 1.2 需求/设计类

| # | 卡点 | 现象 | 根因 | 影响 |
|:-:|------|------|------|------|
| D1 | 前端缺少登录流程 | 后端所有接口要 JWT，但前端模板没考虑认证 | 技术设计阶段没有把"认证"作为横切关注点纳入前端 scope | 联调时临时补登录页 |
| D2 | 没有 Register 接口 | 测试时只能手动往 DB 插用户 | 技术设计拆任务时没考虑"测试数据初始化" | 测试效率低 |
| D3 | 迭代文档与代码不一致 | `tech-design.md` 里的接口签名和实际代码对不上 | Phase 3 开发时没有强制"改完代码立即同步迭代文档" | 归档前必须逐个比对修复 |

### 1.3 流程协作类

| # | 卡点 | 现象 | 根因 | 影响 |
|:-:|------|------|------|------|
| F1 | 阶段转换等指令 | Phase 3 做完等"commit 吧"，联调完等"归档吧" | skill.md 没有定义"满足 X 条件就自动进入下一步"的触发机制 | 用户介入 10 次，6 次不必要 |
| F2 | trivial fixes 还在确认 | 密码不对这种明显错误还在问用户要不要改 | Agent 倾向于保守，宁可多问也不犯错 | 用户说了"不要问我，直接改" |
| F3 | 前后端多仓库协调 | commit 要分别操作两个仓库，容易遗漏 | Harness 初始化时没考虑多仓库场景 | 操作繁琐 |

### 1.4 归档类

| # | 卡点 | 现象 | 根因 | 影响 |
|:-:|------|------|------|------|
| A1 | 归档流程偏重 | archive-checklist.md 有 6 个阶段 30+ 检查项，大部分是复制粘贴 | 归档是低频操作，模板没为"首次使用"做简化 | 第一次走流程需要来回查文档 |
| A2 | 文档合并手工操作多 | 读迭代文档 → 合并到 project/，本质是复制粘贴 | 没有自动化脚本辅助 | 效率低，容易遗漏 |

## 2. 优化方案

### 2.1 环境配置标准化（解决 E1/E3）

**改造内容**：

在 `.harness/` 下新增 `dev-env.yaml`，记录所有本地依赖的连接信息：

```yaml
# .harness/dev-env.yaml
services:
  mysql:
    host: localhost
    port: 3306
    username: root
    password: root123
    database: user_management
    docker_container: harness-mysql
  redis:
    host: localhost
    port: 6379
    docker_container: harness-redis

maven:
  test_skip_flag: "-Dmaven.test.skip=true"  # 跳过编译+执行
  test_compile_skip_flag: "-DskipTests"     # 只跳过执行
```

**执行时机**：Phase 3 开发时，Agent 自动读取 `dev-env.yaml` 生成 `application.yml`，避免密码散落在多处。

**预期收益**：环境配置问题从"3 轮试错"变成"读文件即得"。

---

### 2.2 横切关注点清单（解决 D1/D2）

**改造内容**：

在 Phase 2 技术设计时，增加一个"横切关注点 checklist"：

```markdown
## 横切关注点（Phase 2 必须确认）

- [ ] **认证**：接口是否需要 JWT？前端是否需要登录页？
- [ ] **日志**：关键操作是否需要记录日志？
- [ ] **异常处理**：统一异常返回格式？
- [ ] **测试数据初始化**：是否需要 Register 接口？还是手动插数据？
- [ ] **多端协调**：前后端是否在同一仓库？commit 策略是什么？
```

**执行时机**：Phase 2 技术设计完成后，Agent 自检 checklist，未覆盖的项主动补充到 tech-design.md。

**预期收益**：联调时不会再出现"前端没登录页"这种低级问题。

---

### 2.3 文档同步前置（解决 D3）

**改造内容**：

把文档同步从"归档时做"改成"每个切片完成时做"。

**切片完成 checklist**：

```markdown
## 切片完成 checklist

- [ ] 代码实现 ✅
- [ ] 单元测试通过 ✅
- [ ] **文档同步** ✅
  - [ ] `api-changes.md`：新增/修改的接口已同步
  - [ ] `ddl-changes.md`：新增/修改的表已同步
  - [ ] `tech-design.md`：接口签名和实际代码一致
```

**执行时机**：Phase 3 每个切片完成后，Agent 自检 checklist，不通过则不进入下一切片。

**预期收益**：归档时不再需要逐个比对修复文档。

---

### 2.4 Agent 决策标准矩阵（解决 F1/F2）

**改造内容**：

在 skill.md 里增加明确的决策矩阵，定义 Agent 什么自决、什么必问：

| 决策类型 | Agent 行为 | 理由 |
|---------|:---------:|------|
| **代码级修复**（编译错误、import 缺失、配置不匹配） | **自决** | 不影响架构，改完跑测试验证即可 |
| **阶段自动推进**（切片完成 → 下一切片，联调通过 → 归档） | **自决** | 流程内推进，不需要人按"下一步" |
| **Git 操作**（commit / push） | **自决** | 每个阶段结束自动 commit，Harness 铁律 |
| **技术选型**（框架选择、数据库方案、架构模式） | **必问** | 有长期影响，改了成本很高 |
| **需求变更**（砍功能、加功能、改优先级） | **必问** | 产品决策不是 Agent 的职责 |
| **异常兜底**（连续 3 次自修失败、环境问题搞不定） | **必问** | 真正的卡点，Agent 已经到极限了 |
| **用户验收**（联调结果、最终产出） | **必问** | 人是最终验收者 |

**一句话原则：能跑测试验证的事 Agent 自己干，不能跑测试验证的事问人。**

**执行时机**：Agent 每次要做决策时，先查这个矩阵。

**预期收益**：用户介入从 10 次降到 4 次（只有验收、技术选型、需求变更、异常兜底才需要）。

---

### 2.5 归档流程轻量化（解决 A1/A2）

**改造内容**：

区分"首次归档"和"后续归档"，提供简化版 checklist：

**简化版归档 checklist**：

```markdown
## 归档 checklist（简化版）

### 1. 前置检查
- [ ] `_meta.yaml` status: archived
- [ ] `tasks.md` 所有任务 ✅

### 2. 文档合并（自动）
- [ ] DDL → `project/data-model.md`（追加）
- [ ] API → `project/api-contracts.md`（追加）
- [ ] 模块 → `project/modules/*.md`（更新）

### 3. 目录迁移
- [ ] `mv iterations/{迭代名} archive/{迭代名}`

### 4. AI Context 同步
- [ ] 更新 `AI-CONTEXT.md` 模块状态
- [ ] 追加迭代历史

### 5. Git 提交
- [ ] `git add Docs/ && git commit && git push`
```

**长期目标**：提供归档脚本 `harness-archive.sh`，自动完成目录移动和 AI-CONTEXT 更新。

**预期收益**：归档时间从 ~20 分钟降到 ~5 分钟。

---

### 2.6 多仓库协调机制（解决 F3）

**改造内容**：

在 `.harness/` 下新增 `repos.yaml`，记录多仓库的协调策略：

```yaml
# .harness/repos.yaml
repos:
  - name: user-management
    path: ~/user-management
    type: backend
    auto_commit: true
  - name: user-management-frontend
    path: ~/user-management-frontend
    type: frontend
    auto_commit: true

commit_strategy:
  on_phase_complete: all_repos  # 阶段完成时所有仓库 commit
  commit_message_prefix: "[{phase}] "
```

**执行时机**：Phase 完成时，Agent 遍历所有仓库，自动 commit。

**预期收益**：多仓库 commit 从"分别操作"变成"一键提交"。

---

### 2.7 迭代文档模板增加"已知限制"（预防未来卡点）

**改造内容**：

在 `_meta.yaml` 里增加 `known_limitations` 字段：

```yaml
known_limitations:
  - 模拟支付，未对接真实支付渠道
  - 无 Register 接口，测试用户需手动插入 DB
  - 前端未实现错误重试机制
```

**执行时机**：Phase 3 开发时，Agent 发现临时方案就记录到 `known_limitations`。

**预期收益**：后续迭代知道哪里要补，不用重新排查。

## 3. 优先级排序

| 优先级 | 改造项 | 理由 | 预估工时 |
|--------|--------|------|---------|
| **P0** | Agent 决策标准矩阵 (2.4) | 解决用户介入过多问题，影响最大 | 2h |
| **P0** | 环境配置标准化 (2.1) | 解决最高频的卡点（密码试错） | 1h |
| **P1** | 文档同步前置 (2.3) | 解决归档时的文档不一致问题 | 2h |
| **P1** | 横切关注点清单 (2.2) | 预防联调时的低级问题 | 1h |
| **P2** | 归档流程轻量化 (2.5) | 低频操作，但能提升效率 | 3h（含脚本） |
| **P2** | 多仓库协调 (2.6) | 多仓库场景才需要 | 2h |
| **P3** | 已知限制字段 (2.7) | 锦上添花 | 0.5h |

**建议执行顺序**：P0 → P1 → P2 → P3，总计 ~12h。

## 4. 验收标准

改造完成后，用下一个迭代验证：

| 验收项 | 标准 |
|--------|------|
| 用户介入次数 | 从 10 次降到 ≤ 5 次 |
| 环境配置时间 | 从 ~10 分钟降到 ~1 分钟 |
| 文档一致性 | 归档时无需修复文档不一致 |
| 联调低级问题 | 无"前端没登录页"类问题 |
| 归档时间 | 从 ~20 分钟降到 ~5 分钟 |

---

## 5. 不做的事

- **不做自动化编排器**：当前阶段用方法论指导人工编排，摸清卡点后再考虑自动化
- **不做状态机**：不用复杂的有限状态机描述流程，用简单的 checklist 即可
- **不做 Agent 自循环**：Agent 不需要自己推进到下一阶段，满足条件就自动进入即可
