# 快速反馈

## 要解决的问题

AI 写一行代码，多久能知道它对不对？

Python 项目通常是解释执行即时报错，Java 项目却要走：
```
改代码 → 编译 → 注解处理 → Spring 启动 → 配置绑定 → 依赖初始化 → HTTP 调用 → 验证
```

这 10~30 秒的延迟对 AI 是致命的——AI 需要高频试错，等待成本吞掉效率红利。

## 核心原则

**把验证链路切成多层"短回路"，不让 AI 每次改一行等 30 秒。**

### 三层反馈

| 层 | 验证内容 | 目标时延 | 工具（技术无关描述） |
|----|---------|:---:|------|
| 语法+依赖 | 能否编译 | ≤ 3s | 增量编译 |
| 装配+核心行为 | 模块能否启动、核心逻辑对不对 | ≤ 30s | 模块切片测试 |
| 完整集成 | 跨模块、真实依赖是否正常 | 按需 | 集成测试 |

### 关键决策

- **"完整启动"是特权，不是默认。** 默认只启动被修改模块的最小装配。
- **慢测试和快测试物理隔离。** 不要让一次完整集成测试阻塞你 100 次快速迭代。
- **编译缓存是基础设施，不是优化项。** 项目第一天就应该配好。

## 与技术栈无关的检查清单

- [ ] 是否有增量编译？
- [ ] 是否有"只验证当前模块"的测试入口？
- [ ] 快测试和慢测试是否分开？
- [ ] 完整启动是否是必要动作而不是默认动作？
- [ ] 最常用的验证路径是否在 30 秒内？
