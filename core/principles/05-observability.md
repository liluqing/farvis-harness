# 可观测性

## 要解决的问题

AI 不怕报错，怕的是"报错没有上下文"。

```
❌ NullPointerException
❌ SQL 慢但没有参数
❌ 日志没有 traceId
❌ 线程池满了但不知道队列长度
```

AI 看到的只是"现象"；有了可观测性，AI 才能看到"原因链"。

## 核心原则

**让 AI 拿到的不只是报错信息，而是定位问题所需的完整上下文。**

### 三件套

| 组件 | 回答的问题 | 最低要求 |
|------|----------|---------|
| **结构化日志** | 这次请求发生了什么？ | JSON 格式 + traceId + 关键业务字段 |
| **业务指标** | 系统整体健康吗？ | Counter（成功/失败/幂等命中）+ Timer（P99） |
| **分布式追踪** | 这次请求经过了哪些服务？哪一步慢了？ | traceId 贯穿所有日志 + span |

### 不只是运维的事

可观测性对 AI 开发的直接价值：

1. **AI 修 bug 时**：能看到完整的请求链路，不是孤立的异常堆栈
2. **AI 优化性能时**：能看到 P99 延迟分布，知道瓶颈在哪一层
3. **AI 写新代码时**：知道日志和指标的上报规范，写的代码自带可观测性

### Error Catalog：把组织经验结构化

```
errors:
  DUPLICATE_REQUEST:
    cause: "重复 requestId 导致幂等命中"
    action: "返回已有结果，不重复处理"
    log_level: "INFO"
```

每遇到一个线上问题，就应该在 error-catalog 里加一条。这是"组织经验"变成"AI 知识"的通道。

## 与技术栈无关的检查清单

- [ ] 日志是否是结构化的（JSON），而不是字符串拼接？
- [ ] 每条关键日志是否包含 traceId？
- [ ] 核心业务操作是否有 Metrics？
- [ ] 是否有 Error Catalog（不只是代码里的错误码枚举）？
- [ ] 出问题时，能否 2 分钟内定位到具体模块/依赖/数据层？
