# 上下文契约

## 要解决的问题

AI 缺的不是代码行数，是**对项目结构的理解**。

你把 500 个 .java 文件喂给 AI，它看到的是碎片。真正决定代码质量的——模块边界、幂等策略、缓存约定、错误处理规范——散落在团队老员工的脑子里、Wiki 的角落、口口相传的习惯里。

## 核心原则

**把团队的隐式知识，结构化地暴露给 AI。**

### 五类上下文资产

| 资产 | 回答的问题 | 格式 |
|------|----------|------|
| **Project Map** | 这个项目有几个模块？谁依赖谁？谁不能碰谁？ | YAML：模块名→职责→对外接口→禁止项 |
| **Business Rules** | 这个操作需要幂等吗？缓存多久？事务边界在哪？ | YAML：操作名→约束列表 |
| **Coding Rules** | Controller 能直接调 Mapper 吗？异常怎么处理？ | YAML：规则列表 |
| **Error Catalog** | 这个错误码是什么意思？出错了该怎么办？ | YAML：错误码→原因→修复路径 |
| **ADR** | 为什么选这个技术？当时还考虑了什么？ | Markdown：决策记录 |

### 与 Prompt Engineering 的区别

Prompt 告诉你"怎么写代码"。上下文契约告诉你"在这个项目里怎么写是对的"。

一个 500 字的 coding-rules.yaml 比 5000 字的 prompt 更有效——因为它是结构化的、可验证的、可自动检查的。

## 与技术栈无关的检查清单

- [ ] 新成员（包括 AI）能否 5 分钟内了解模块边界？
- [ ] 业务规则是否结构化，不只是注释或 Wiki？
- [ ] 编码约定是否机器可检查？
- [ ] 常见错误的修复路径是否有文档？
- [ ] 上下文资产是否自动生成，而非全依赖人工维护？
