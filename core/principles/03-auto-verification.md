# 自动验证

## 要解决的问题

AI 写得快，不代表系统能信它。

最大的组织风险不是"代码写错了"，而是"代码看起来没问题，于是被过快合入"。自动验证把"主观相信"替换为"客观校验"。

## 核心原则

**按业务风险组织测试，不按技术层组织测试。**

### 四层验证

| 层 | 验证什么 | 运行频率 | 示例 |
|----|---------|:---:|------|
| 编译+静态规则 | 语法 + 架构约束 | 每次保存 | 分层依赖校验 |
| 业务单元测试 | 核心逻辑正确性 | 每次修改 | 幂等逻辑、状态机 |
| 基础设施集成 | 与真实 DB/缓存/队列的兼容性 | PR 提交 | SQL、事务、序列化 |
| 契约测试 | 跨服务边界不被破坏 | PR 合并 | API 协议、事件 schema |

### 测试的组织方式

不是：
```
test/
  unit/     ← 验证 getter/setter，价值低
  integration/
```

而是：
```
test/
  risk-idempotency/   ← 所有幂等敏感的操作
  risk-concurrency/   ← 所有并发敏感的操作
  risk-degradation/   ← 所有降级路径
```

### 关键业务风险用例模式

所有 Java 项目都应该测试这几类风险：

| 类别 | 问题 | 测试方法 |
|------|------|---------|
| 幂等 | 重复请求是否只生效一次？ | 多线程同 key 并发提交 |
| 乱序 | 回调/消息乱序到达时状态机是否正确？ | 反向顺序触发事件 |
| 降级 | 下游不可用时是否安全降级？ | 模拟超时/5xx |
| 一致性 | 失败时是否完整回滚/退款？ | 模拟中间步骤失败 |
| 并发 | 乐观锁冲突时是否安全重试？ | 高并发争用同一资源 |

## 与技术栈无关的检查清单

- [ ] 测试是按业务风险还是按技术层组织的？
- [ ] 幂等操作是否有并发测试？
- [ ] 下游不可用时是否有降级测试？
- [ ] 关键状态机是否有乱序测试？
- [ ] 涉及外部 API 的测试是否依赖真实服务？（应该不依赖）
