# 幂等模式

## 问题

网络重试、客户端重复提交、消息重复投递——同一个请求可能被处理多次。

如果没有幂等保护：重复扣费、重复创建、数据错乱。

## 模式

**基于"幂等键"的去重机制。**

```
1. 客户端生成唯一 requestId
2. 服务端收到请求：
   ├── 查幂等键是否已存在
   │   ├── 存在 → 返回已有结果（不重复处理）
   │   └── 不存在 → 执行业务 → 记录幂等键 → 返回结果
```

## 关键决策

| 决策点 | 选项 | 选型考虑 |
|--------|------|---------|
| 幂等键来源 | 客户端生成 / 业务字段组合 | 客户端生成最可靠 |
| 幂等键存储 | Redis（TTL 自动过期）/ DB 唯一索引 | Redis 性能最好，DB 索引最可靠 |
| 幂等键有效期 | 10 分钟 / 1 小时 / 24 小时 | 取决于业务重试窗口 |
| 幂等粒度 | 请求级 / 业务操作级 | 通常按请求级（requestId） |

## 并发场景

多线程同时到达同一幂等键时：

### Redis 方案
```
SETNX idempotency-key:{requestId} → 只有一个线程成功
失败的线程 → GET 已有结果 → 返回
```

### DB 唯一索引方案
```
INSERT (requestId, ...) ON DUPLICATE KEY → 报唯一约束冲突
冲突 → SELECT 已有记录 → 返回
```

## 与技术栈无关的约束

- 幂等键必须由客户端生成并传入
- 幂等键必须包含在业务规则清单中（ai-context/business-rules.yaml）
- 幂等逻辑必须测试并发场景（≥ 2 线程）
