# 熔断模式

## 问题

下游服务抖动时，如果不加保护：

```
请求 → 调下游（超时 30s）→ 线程阻塞 → 线程池耗尽 → 整个服务不可用
```

一个慢的下游可以拖垮整个系统。

## 模式

**快速失败 > 缓慢超时。**

当下游连续失败达到阈值时，熔断器打开——后续请求直接失败，不再等待超时。一段时间后进入半开状态，探测下游是否恢复。

### 三态状态机

```
          连续失败 N 次
  CLOSED ────────────────▶ OPEN
    ▲                       │
    │                       │ 等待 T 秒
    │                       ▼
    └───────────────── HALF_OPEN
           探测成功            探测失败 → 回到 OPEN
```

## 关键决策

| 决策点 | 建议值 | 说明 |
|--------|:---:|------|
| 失败阈值 | 5 次 | 连续 5 次失败后打开 |
| 半开等待 | 30s | 30 秒后尝试探测 |
| 半开探测请求数 | 1~3 个 | 少量请求探测 |
| 超时计入失败 | 是 | 超时也计为失败 |
| 熔断后行为 | 抛出异常 / 返回降级结果 | 取决于业务 |

## 降级策略

熔断打开后，不是只抛异常，而是要有降级：

| 场景 | 降级方案 |
|------|---------|
| 风控服务不可用 | 默认放行（标记待复核） |
| 推荐服务不可用 | 返回热门列表 |
| 支付服务不可用 | 提示用户稍后重试 + 保留订单 |
| 外部 API 不可用 | 返回缓存数据（标记过期） |

## 与技术栈无关的约束

- 所有外部 API 调用必须配置超时 + 熔断
- 熔断打开时必须有降级策略（不能只抛异常）
- 熔断事件必须上报 Metrics
